### 3.3 关键技术决策

**决策 1：注册端点角色强制策略**
- **问题**：原实现允许客户端传入任意 role 字段，可注册为 FOUNDER
- **方案**：端点内强制 `role=UserRole.RESEARCHER`，忽略 payload.role
- **理由**：公开注册端点不应允许权限自选；提权操作必须由 FOUNDER 通过 `/users/{id}/role` 完成
- **测试影响**：conftest.py 的 auth_token fixture 改为直接在 DB 创建 founder 用户

**决策 2：JWT 密钥分级校验**
- **问题**：默认密钥为公开字符串，攻击者可伪造任意 token
- **方案**：field_validator 区分环境 — 生产环境拒绝默认值且要求 ≥32 字节，开发环境记录警告
- **理由**：平衡开发便利性与生产安全性，避免 pydantic_settings 加载顺序问题

**决策 3：加密降级策略**
- **问题**：无密钥时 encrypt() 静默返回明文，掩盖未加密事实
- **方案**：生产环境抛 RuntimeError，开发环境记录 warning 并返回明文
- **理由**：decrypt() 保持容错（避免密钥轮换期间业务中断），encrypt() 严格（防止明文落库）

**决策 4：文件上传安全加固**
- **问题**：无大小限制可 OOM，无类型校验可上传恶意文件
- **方案**：MAX_UPLOAD_BYTES=100MB + ALLOWED_EXTENSIONS 白名单 + asyncio.to_thread 异步写入
- **理由**：100MB 覆盖常规组学数据文件；白名单防止可执行文件；异步写入避免阻塞事件循环

## 四、契约不一致点记录

### 4.1 列表端点返回原始 List（非信封）
- **现状**：projects/data/targets/molecules 等 7 个列表端点返回 `List[Model]`，未经 PagedResponse 包装
- **影响**：前端无法统一解析分页 meta，X-Request-ID 不在 body 中
- **建议**：后续迭代统一为 `paged_response(data, page, page_size, total, ...)`

### 4.2 部分端点无 GET /{id} 详情路由
- **现状**：molecules/treatments/experiments 端点未实现详情路由，返回 FastAPI 默认 `{"detail": "Not Found"}`
- **影响**：错误响应不符合 ErrorResponse 契约
- **建议**：补全详情路由或统一 404 处理器

### 4.3 成功响应 body.meta.request_id 为空
- **现状**：部分端点调用 `success_response(request_id="")`，未从请求上下文获取 request_id
- **影响**：body.meta.request_id 与 X-Request-ID 头不一致
- **建议**：端点使用 `get_request_id()` 从上下文获取，或中间件统一注入

## 十、P5 后续：水平越权校验（行级安全）（2026-07-08）

### 10.1 背景与目标

P4 安全审查识别出 P2 级漏洞"水平越权（无 owner_id 校验）"：列表端点返回全部数据，
任何认证用户均可遍历全组织的项目、数据集、靶点等敏感资源。本次修复为 7 个列表端点
添加行级安全（Row-Level Security）过滤，确保非领导角色仅可见自己拥有的项目及其关联资源。

### 10.2 可见性规则

依据 `ROLE_PERMISSIONS` 矩阵定义的角色权限范围：

| 角色 | 权限范围 | 列表可见性 |
|------|----------|------------|
| FOUNDER | admin:all / data:read | 全组织可见 |
| CHIEF_RESEARCHER | data:read / analysis:read | 全组织可见（研究领导层） |
| RESEARCHER | data:read:assigned | 仅自己拥有的项目及其关联资源 |
| DOCTOR | data:read:clinical | 仅自己拥有的项目（临床上下文） |
| DATA_ENGINEER | system:logs / quality:read | 仅自己拥有的项目 |

### 10.3 资源归属链

| 资源 | 归属路径 | 过滤方式 |
|------|----------|----------|
| Project | owner_id（直接） | `Project.owner_id == current_user.id` |
| Dataset | project_id → Project.owner_id | 子查询可见项目 ID |
| Target | project_id → Project.owner_id | 子查询可见项目 ID |
| Hypothesis | project_id → Project.owner_id | 子查询可见项目 ID |

1. **子查询而非 JOIN**：保持 SELECT 子句不变，兼容 `select(func.count()).select_from(...)` 计数查询

### 11.2 slowapi 登录限流

**背景**：登录端点无暴力破解防护，攻击者可无限尝试密码。

**变更**：
1. **新增 `app/core/limiter.py`** — slowapi Limiter 实例，按客户端 IP 计数
2. **`/auth/login` 端点限流**：`@limiter.limit("5/minute")`，每个 IP 每分钟最多 5 次登录尝试
3. **配置项**：`LOGIN_RATE_LIMIT_PER_MINUTE: int = 5`（始终启用，独立于通用 `RATE_LIMIT_ENABLED`）
4. **429 响应统一信封格式**：自定义 `_rate_limit_envelope_handler`，返回 `{"success": false, "error": {"code": "RATE_LIMITED", ...}}`
5. **中间件顺序调整**：EnvelopeMiddleware 在 SlowAPIMiddleware 内部，避免 BaseHTTPMiddleware 的 body 分块破坏信封注入

**关键技术决策**：
- **中间件顺序**：SlowAPIMiddleware 基于 BaseHTTPMiddleware，会将响应体分块发送（`more_body=True`），导致 EnvelopeMiddleware 误判为流式响应而不注入 `duration_ms`。将 EnvelopeMiddleware 放在 SlowAPIMiddleware 内部，使其在 body 被分块前完成注入。
- **429 响应格式**：slowapi 默认返回 `{"error": "..."}`，不符合统一信封规范。自定义 handler 包装为 `{"success": false, "error": {"code": "RATE_LIMITED"}, ...}`。
- **测试隔离**：conftest.py 新增 autouse fixture，每个测试前清空 limiter 存储避免测试间相互影响。

### 13.4 确认的 Bug：target_identifier.py L88 None 值迭代 TypeError

**Bug 位置**：`backend/app/services/analyzer/target_identifier.py` L87-90

**Bug 描述**：
```python
for marker_cluster in (summary.get("top_markers_per_cluster") or {}).values():
    for m in marker_cluster:  # ← marker_cluster 可能为 None
        if isinstance(m, dict) and m.get("gene"):
            diff_genes.append(m["gene"])
```

当 `top_markers_per_cluster` 字典的某个值是 None 时，`for m in None` 抛出 `TypeError: 'NoneType' object is not iterable`。

**根因分析**：
- `parsed_summary` 是 Dataset 模型的 JSON 字段，存储在数据库中
- scrna.py 生成的 `top_markers_per_cluster` 值始终是列表（L82 初始化为 `{}`，L90-93 由列表推导式生成）
- 但数据库 JSON 字段可被外部修改（API、数据库直接操作、数据导入），导致值可能为 None
- 代码仅对 `top_markers_per_cluster` 本身做了 `or {}` 兜底，未对字典值做类型校验

**影响等级**：P2
- 不会导致 500 错误（orchestrator.py L257-259 有外层 try/except 捕获）
- 但会导致靶点发现功能降级为纯 LLM 模式，丢失多组学数据驱动的靶点候选
- 影响数据完整性：差异基因列表为空，confidence_score 计算失真

### 13.6 修复方案与实施

**修复方案**：在 `for m in marker_cluster:` 前添加 `isinstance` 类型检查，跳过非列表值。

**修复代码**（`target_identifier.py` L87-92）：
```python
for marker_cluster in (summary.get("top_markers_per_cluster") or {}).values():
    if not isinstance(marker_cluster, list):  # 新增：跳过 None/非列表值
        continue
    for m in marker_cluster:
        if isinstance(m, dict) and m.get("gene"):
            diff_genes.append(m["gene"])
```

**设计决策**：
1. **使用 `isinstance` 而非 `is not None`**：`isinstance` 更严格，确保仅处理列表类型，避免字符串、字典等可迭代但语义错误的值
2. **`continue` 而非抛出异常**：跳过无效数据，保证部分数据可用（防御性编程）
3. **符合项目技术规范**：与同文件 L81-85 对 `variants` 字段的处理方式一致（`isinstance(v, str)` / `isinstance(v, dict)`）
