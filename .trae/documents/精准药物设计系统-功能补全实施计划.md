## Context（背景与动机）

参照 `g:\软件开发\AI药物\.qoder\repowiki` 中的产品设计规范，对现有系统进行全面功能对照与补全。已通过 3 份 Explore Agent 差异分析报告确认：当前实现（v1.0.0）相比设计规范缺失约 5 个完整 API 模块、10 个服务、8 个数据模型、3 个前端页面、6+ 个组件，且现有 16 个端点与 9 个页面存在功能缺口。用户已确认三大决策：**全量分阶段实施**、**全面改造为统一信封**、**路径对齐设计规范**。本计划目标是将系统从"框架级 v1.0.0"提升到"功能完整 v1.1.0"，使其与设计规范匹配度 ≥95%。

## 阶段 P0：后端核心基础设施

### 目标
建立统一响应信封、异常体系、信封中间件，为所有后续模块提供基础设施。**P0 是所有后续阶段的前置依赖**。

### 任务

#### P0.1 统一响应信封与元数据
- 新建 `backend/app/schemas/common.py`：
  - `ResponseMeta{request_id: str, duration_ms: Optional[int]}`
  - `ApiResponse[T]{success: bool=True, data: Optional[T], meta: ResponseMeta}`（泛型 `TypeVar`）
  - `PagedResponse[T]{success: bool=True, data: List[T], meta: ResponseMeta + {page, page_size, total, total_pages}}`
- 修改 `backend/app/api/v1/schemas/__init__.py`：
  - `StandardResponse` 改为 `ApiResponse[Any]` 别名（保兼容）
  - 现有 schema 保持不变

#### P0.2 异常体系
- 新建 `backend/app/core/exceptions.py`：
  - `AppException(code, message, details, status_code)` 基类
  - 8 个子类：`ValidationError(400, VALIDATION_ERROR)`、`UnauthorizedError(401, UNAUTHORIZED)`、`ForbiddenError(403, FORBIDDEN)`、`NotFoundError(404, NOT_FOUND)`、`ConflictError(409, CONFLICT)`、`GuardrailBlockedError(422, GUARDRAIL_BLOCKED)`、`RateLimitedError(429, RATE_LIMITED)`、`UpstreamError(502, UPSTREAM_ERROR)`
  - `_error_envelope(exc, request_id) -> dict` 返回 `{success:false, error:{code, message, details}, meta:{request_id}}`
  - `register_exception_handlers(app)` 注册 AppException、RequestValidationError、HTTPException、Exception 兜底

#### P0.3 信封中间件
- 新建 `backend/app/core/middleware.py`：
  - `EnvelopeMiddleware(BaseHTTPMiddleware)`：
    - 解析或生成 `X-Request-ID`，写入 `scope["state"]["request_id"]` 与响应头
    - 计算耗时，写入 `X-Response-Time-ms` 响应头
    - 仅对 `application/json` 且 `content-length < 1MB` 的 200 响应注入 `meta.duration_ms` 并修正 content-length
    - 流式响应（`more_body=True`）直接透传
  - `get_request_id() -> str` 基于 `contextvars` 的 helper，供服务层日志关联

#### P0.4 配置扩展
- 修改 `backend/app/core/config.py` 追加配置项（全部带默认值，向后兼容）：
  - LLM：`LLM_DAILY_BUDGET_USD=50.0`、`LLM_COST_TRACKER_REDIS_URL=""`、`GUARDRAIL_ENABLED=True`
  - DP：`DP_DEFAULT_EPSILON=1.0`、`DP_DEFAULT_DELTA=1e-5`
  - FL：`FL_NUM_ROUNDS_DEFAULT=10`、`FL_MIN_CLIENTS_DEFAULT=3`、`FL_MAD_THRESHOLD=3.0`
  - 限流：`RATE_LIMIT_ENABLED=False`、`RATE_LIMIT_RPM=60`、`RATE_LIMIT_STRATEGY="sliding_window"`
  - 中间件：`ENVELOPE_MIDDLEWARE_ENABLED=True`
  - 日志：`LOG_LEVEL="INFO"`、`LOG_FILE_PATH="logs"`、`LOG_ROTATION="10 MB"`

#### P0.5 应用入口改造
- 修改 `backend/app/main.py`：
  - CORS 增 `expose_headers=["X-Request-ID", "X-Response-Time-ms"]`
  - 在 CORS 之前 `add_middleware(EnvelopeMiddleware)`（受 `ENVELOPE_MIDDLEWARE_ENABLED` 开关）
  - 用 `register_exception_handlers(app)` 替换内联 `global_exception_handler`
  - `/health` 迁移到 `api/v1/endpoints/admin.py` 的 `/health`（含 DB/Redis/ChromaDB 探活 + 5s 内存缓存）
  - 根路径 `/health` 保留 1 版本作为别名
  - 新增 `/api/v1/metrics` Prometheus 文本端点

### 验证方式
- `pytest tests/test_envelope.py`：验证 200/400/401/403/404/409/422/429/502/500 全部返回信封结构、`meta.duration_ms` 非空、`X-Request-ID` 透传
- `curl /api/v1/health` 返回依赖健康度 JSON
- `curl /api/v1/metrics` 返回 Prometheus 文本
- 现有 62 个后端测试全绿（确认 `StandardResponse` 别名生效）

### 风险
- **高**：EnvelopeMiddleware 缓冲所有 body 增加内存与延迟 — 缓解：仅 `application/json` + `<1MB` 启用注入，大响应/流式透传
- **中**：现有端点返回 `List[Model]`，信封后前端取数路径变化 — 缓解：P0 后端双轨（`response_model=ApiResponse`，旧端点保持裸 List 直到 P2 完成）

### 关键决策
- **D1**：`/health` 根路径别名保留 1 版本周期，P4 后移除
- **D6**：`StandardResponse` 作为 `ApiResponse[Any]` 别名平滑迁移
- **D8**：信封中间件缓冲阈值 `<1MB` JSON

## 关键决策汇总

| ID | 决策 | 选择 | 理由 |
|---|---|---|---|
| D1 | `/health` 根路径别名 | 保留 1 版本 | 监控系统已配置 |
| D2 | `treatments`/`experiments` 旧路由 | 保留别名 1 版本 | 前端逐步切换 |
| D3 | `knowledge/privacy_layer.py` | re-export 新位置 | 避免 import 断裂 |
| D4 | React Query 推广策略 | 新页面用，旧页面渐进 | 降低回归风险 |
| D5 | `admin/page.tsx` 拆分 | 拆为 admin + admin/monitoring | 职责分离 |
| D6 | `StandardResponse` | 作为 `ApiResponse[Any]` 别名 | 平滑迁移 |
| D7 | Alembic 引入时机 | P1 末尾 | 新模型多，create_all 不可持续 |
| D8 | 信封中间件缓冲阈值 | `<1MB` JSON | 平衡内存与功能 |

## 风险与缓解

| 风险 | 等级 | 缓解 |
|---|---|---|
| 信封改造破坏 16 现有端点前端消费 | 高 | P0 后端双轨 + P2 拦截器统一解包 + 旧路径别名 |
| LLM 接入 Router/Guardrail 改变 chat 行为 | 高 | 保留 `tier` 参数 + Guardrail 返回 422 + 灰度开关 `GUARDRAIL_ENABLED` |
| 9 现有页面同时改造引入回归 | 高 | 逐页独立提交 + 视觉回归 + E2E |
| Alembic 与 create_all 数据库冲突 | 中 | env.py 检测后 `stamp head` |
| RDKit.js/3Dmol.js 体积 | 中 | dynamic import + ssr:false |
| 新增服务依赖 Redis（cost_tracker） | 中 | Redis 不可用时降级内存计数 + 告警 |
| 联邦学习/隐私计算真实环境验证 | 低 | Mock 模式下完整流程可跑通，Real 模式标记 experimental |

## 实施顺序与并行性

```
P0（schemas ← exceptions ← middleware ← config ← main）
  ↓
P1（models ← services ← endpoints ← router ← alembic）
  ↓                          ↑ 可并行
P2（api.ts ← queryClient ← ui 组件 ← charts 组件）
  ↓
P3（新页面 ← 现有页面补全 ← Sidebar）
  ↓
P4（测试 ← 契约 ← 旧路径移除）
```

P0 完成后，P1（后端）与 P2 的组件库部分可并行；P2 的 API 改造需等 P1 端点稳定；P3 必须在 P2 完成后；P4 贯穿全程但集中在末尾收尾。
