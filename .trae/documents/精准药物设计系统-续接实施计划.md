## 实施步骤

### 步骤 1：LLM 编排服务（步骤 C 剩余，4 个文件）

**目标**：让 `/chat` 和 `/chat/analyze` 端到端可用，实现分级路由（fast_screen / deep_insight）。

#### `backend/app/services/llm/__init__.py`
空包初始化文件。

#### `backend/app/services/llm/prompts.py`
- 常量字典 `SYSTEM_PROMPTS`：
  - `"fast_screen"`: "你是精准医学快速筛查助手，基于规则和已知数据库快速回答，回答简洁..."
  - `"deep_insight"`: "你是精准医学深度分析专家，请综合文献、变异注释、通路信息、PPI 网络进行深度分析，给出机制级解释..."
- 常量 `TARGET_DISCOVERY_PROMPT`、`MOLECULE_DESIGN_PROMPT`、`EVIDENCE_CHAIN_PROMPT`、`REPURPOSING_PROMPT`
- 函数 `build_context_prompt(context: dict) -> str`：将基因/变异/药物/通路等上下文拼装为结构化 prompt

#### `backend/app/services/llm/rag.py`
- 类 `RAGEngine`
- `__init__(self, db, llm_client)`
- `async def retrieve(query: str, project_id: str = None, top_k: int = 5) -> List[dict]`：
  - 调 `VectorStore.search(query, collection=f"project_{project_id}", top_k)`
  - Mock 模式或 ChromaDB 不可用时返回空列表（不阻塞）
- `async def augment(query: str, retrieved: List[dict]) -> str`：
  - 将检索到的文档片段拼入 prompt 上下文
  - 无检索结果时返回原始 query

#### `backend/app/services/llm/orchestrator.py` — 分级路由核心
- 类 `LLMOrchestrator`
- `__init__(self, db, llm_client)`：存 db 和 llm_client
- `async def route(message, project_id, tier, user) -> dict`：
  1. 记录 start_time
  2. 根据 `tier` 选择策略：
     - `fast_screen`：直接调 `llm_client.chat(messages)`，system prompt 用 `SYSTEM_PROMPTS["fast_screen"]`，cost 估算按 token 数 ≈$0.01
     - `deep_insight`：调 `RAGEngine.retrieve()` 获取上下文 → 调 `KnowledgeGraph.get_neighbors()` 扩展 → 构建 augment prompt → 调 `llm_client.chat()`，cost 估算 ≈$0.5
  3. 计算 duration_sec 和 cost_usd
  4. 创建 `AnalysisJob` 记录（含 user_id, tier, cost_usd, duration_sec, model, status）
  5. 返回：`{answer, tier, cost_usd, duration_sec, model, references, code}`
  - Mock 模式下 llm_client 返回预置回答，references 为空，code 为 None
- `async def full_analysis(message, project_id, tier, user) -> dict`：
  1. 复现 Sid 团队流程：解析用户意图 → 判断是靶点发现/老药新用/分子设计 → 调对应 Analyzer service → 调 LLM 生成报告 → 生成 Python 分析代码
  2. 步骤：
     - 意图识别（基于关键词：靶点/老药新用/分子/通路）
     - 调 `TargetIdentifier(db).discover(project_id, tier=tier)` 或 `DrugRepurposer(db).repurpose(target)`
     - 调 `llm_client.chat()` 生成结构化报告
     - 生成 Python 代码片段（Mock 模式返回预置分析代码模板）
  3. 返回：`{report, charts: [...], code, references, hypothesis, conclusion}`

### 步骤 2：子系统 C/D 协作平台服务（步骤 D，12 个文件）

**目标**：让 `/experiments`、`/workflows`、`/reports`、`/treatments` 端到端可用。

#### 2a. CDISC 导出（2 个文件）

**`backend/app/services/cdisc/__init__.py`** — 空包

**`backend/app/services/cdisc/sdtm_exporter.py`**
- 类 `SDTMExporter(db)`
- `async def export(project_id) -> dict`：
  - 查询项目的 targets、datasets、experiments、treatments
  - 构建 SDTM 域：
    - `DM`（Demographics）：项目→患者基础信息（USUBJID=project_id, STUDYID, DOMAIN=DM）
    - `VS`（Vital Signs）：数据集元数据（VSTEST=数据类型, VSORRES=文件格式）
    - `RS`（Disease Response）：靶点发现结果（RSTEST=基因符号, RSORRES=证据等级）
    - `EX`（Exposure）：治疗方案（EXTRT=方案名称, EXDOSE=配置）
    - `SV`（Subject Visits）：实验迭代（VISIT=iteration, SVSTDTC=created_at）
  - 返回：`{domains: {DM: [...], VS: [...], RS: [...], EX: [...], SV: [...]}, metadata: {study_id, version, export_time}}`
- `def to_csv(sdtm_data: dict) -> str`：
  - 同步方法（reports.py 中 `exporter.to_csv(sdtm_data)` 直接调用）
  - 将多域数据转为 CSV 字符串，每个域一个 section（`--- DM Domain ---\n` + header + rows）
- `async def export_adam(project_id) -> dict`：
  - 基于 SDTM 派生 ADaM 数据集：
    - `ADSL`（Subject-Level）：从 DM + 汇总
    - `ADRS`（Response Analysis）：从 RS + 疗效
    - `ADAE`（Adverse Events）：从 experiments failed
  - 返回：`{datasets: {ADSL: [...], ADRS: [...], ADAE: [...]}}`

#### 2b. 实验服务（3 个文件）

**`backend/app/services/experiment/__init__.py`** — 空包

**`backend/app/services/experiment/lims_importer.py`**
- 类 `LimsImporter(db)`
- `async def import_data(payload: dict) -> dict`：
  - 解析 LIMS 标准格式 payload：`{experiments: [{name, exp_type, project_id, config, result, lab_source}]}`
  - 批量创建 `Experiment` 记录
  - 返回：`{count, imported_ids: [...], errors: [...]}`

**`backend/app/services/experiment/feedback_loop.py`** — 干湿闭环核心
- 类 `FeedbackLoop(db)`
- `async def apply_feedback(experiment) -> dict`：
  1. 比对 dry prediction（experiment.config 中的 `predicted` 字段）vs wet result（experiment.result 中的 `measured` 字段）
  2. 计算误差：`{mae, rmse, direction_match}`（direction_match = 预测方向与实测方向是否一致）
  3. P3 框架：try import `FederatedLearner`，失败则跳过模型更新
  4. 标记 `experiment.feedback_applied = True`
  5. 创建下一迭代实验建议：基于误差决定是否需要下一轮
  6. 返回：`{feedback: {error_metrics, model_updated, next_iteration, suggested_adjustments}}`

#### 2c. 工作流服务（3 个文件）

**`backend/app/services/workflow/__init__.py`** — 空包

**`backend/app/services/workflow/nextflow_runner.py`**
- 类 `NextflowRunner(db)`
- `async def run(workflow_run) -> dict`：
  1. 根据 `workflow_run.pipeline_name` 选择 nextflow 脚本路径（`nextflow/{pipeline_name}.nf`）
  2. **P0 模拟模式**（默认，`EXECUTE_NEXTFLOW` 环境变量未设或 false）：
     - 不实际执行 nextflow，返回模拟结果
     - status=completed, output_path=`/data/outputs/{run_id}/`
     - duration_sec 随机 30-120
  3. **真实模式**（`EXECUTE_NEXTFLOW=true`）：用 `asyncio.create_subprocess_exec` 执行 `nextflow run {script} -params-file {params_json}`
  4. 更新 `workflow_run.status`、`run_id`、`trace_url`、`output_path`、`duration_sec`
  5. 返回：`{status, run_id, output_path, duration_sec, pipeline_name}`
- `async def check_status(run_id) -> dict`：查询运行状态（P0 返回模拟）

**`backend/app/services/workflow/pipeline_manager.py`**
- 类 `PipelineManager`
- `def list_pipelines() -> List[dict]` — 返回可用管道列表（scrna_pipeline/rna_seq_pipeline/variant_annotation）
- `def get_pipeline_config(name) -> dict` — 返回管道配置（input/output/params 模板）

#### 2d. 优化器服务（4 个文件，均为 P3 框架）

**`backend/app/services/optimizer/__init__.py`** — 空包

**`backend/app/services/optimizer/treatment_planner.py`** — P3 框架
- 类 `TreatmentPlanner(db)`
- `async def optimize(project_id) -> dict`：
  - 查询项目的 targets + treatments
  - P0 框架：基于规则评分（靶点证据等级 + 药物类药性），返回组合建议
  - P3 实现：try import torch RL，失败则降级到规则评分
  - 返回：`{combinations: [{treatments, predicted_efficacy, risk, rationale}], optimal, method}`

**`backend/app/services/optimizer/efficacy_monitor.py`** — P3 框架
- 类 `EfficacyMonitor(db)`
- `async def check(treatment_id) -> dict`：
  - 查询关联的 experiments，汇总结果
  - P0：从 experiment.result 汇总，返回基础指标
  - P3：实时流式监测（框架提示）
  - 返回：`{treatment_id, current_efficacy, trend, adverse_events, recommendation, experiments_count}`

**`backend/app/services/optimizer/federated_learning.py`** — P3 框架
- 类 `FederatedLearner`
- `async def update_weights(local_gradients: dict) -> dict`：P3 Flower 集成入口，try import flower 失败返回框架提示
- `async def aggregate(models: List) -> dict`：FedAvg 算法框架

**`backend/app/services/optimizer/dynamic_adjuster.py`** — P3 框架
- 类 `DynamicAdjuster`
- `async def adjust(treatment_id, efficacy_data) -> dict`：根据疗效动态调整方案，返回建议

**`backend/app/services/knowledge/privacy_layer.py`** — P3 框架
- 类 `PrivacyLayer`
- `async def encrypt_data(data) -> dict`：PySyft 隐私域封装框架
- `async def federated_query(query) -> dict`：联邦查询框架

## Assumptions & Decisions

1. **默认 Mock 模式**：所有外部 API 在 `USE_MOCK=true` 时返回预置数据，无需任何真实 key 即可端到端运行。
2. **P2/P3 框架策略**：DeepChem/PyG/Flower/PySyft 相关代码以 try/except 包裹，import 失败时返回"需启用 P2/P3"提示，不阻塞主流程。
3. **Nextflow 模拟模式**：`NextflowRunner` 默认返回模拟结果，通过 `EXECUTE_NEXTFLOW=true` 切换真实执行。
4. **前端 SSR 安全**：Plotly 用 `dynamic(import, {ssr: false})` 避免 Next.js SSR 报错。
5. **测试用 SQLite in-memory**：避免测试依赖 postgres，conftest 用 sqlite + aiosqlite。
6. **种子数据**：5 个角色演示用户密码统一为 `demo123456`，方便演示。
7. **不修改已存在的 endpoints/models**：所有 service 严格按反推的接口签名实现。
8. **`to_csv` 为同步方法**：因 `reports.py` 中 `exporter.to_csv(sdtm_data)` 在 async 函数内直接同步调用。
9. **`assess_druglikeness` 为同步函数**：因 `molecules.py` 中 `result = assess_druglikeness(smiles)` 直接同步调用（已在 analyzer 中实现）。
10. **前端 API 代理**：通过 next.config.js rewrites 将 `/api/*` 代理到后端，避免 CORS 问题。
