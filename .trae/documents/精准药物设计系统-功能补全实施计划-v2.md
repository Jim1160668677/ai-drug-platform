## 0. 计划摘要

| 阶段 | 名称 | 关键交付物 | 状态 |
|---|---|---|---|
| **P0** | 后端核心基础设施 | 统一响应信封 + 异常体系 + 信封中间件 + 配置扩展 + 入口改造 + 回归测试 | P0.1–P0.3 ✅ / P0.4–P0.6 ⏳ |
| **P1** | 后端业务模块补全 | 5 新模型 + 10+ 新服务 + 5 新端点模块 + 16 现有端点改造 + Alembic | ⏳ |

| **P3** | 前端页面补全 | 4 新页面 + 9 现有页面功能补全 + 侧边栏扩展 | ⏳ |

- 单元测试覆盖率 ≥ 80%
- P0/P1 缺陷 100% 解决
- 保留旧路径 1 个版本周期（双路由别名）
- 所有新增配置项必须带默认值（向后兼容）

## 2. 关键决策（D1–D10）

| ID | 决策 | 理由 |
|---|---|---|
| D1 | `/health` 保留无信封格式，新增 `/api/v1/health` 返回信封格式 | 健康检查常被 K8s/监控直接消费，不宜突变 |
| D2 | `treatments`/`experiments` 端点保留（规范未列但前端已用） | 业务必需，作为扩展端点 |
| D3 | `knowledge`/`privacy_layer` 服务保留并 re-export 到 `services/llm/rag.py`、`services/privacy/privacy_layer.py` | 复用现有实现，避免重复 |
| D4 | 前端新页面用 React Query + 现有 `lib/api.ts` 模式 | 与现有架构一致 |
| D5 | `/admin` 拆分为 Users/LLM/Audit 三个子路由 | 单页过载，拆分提升 UX |
| D6 | `StandardResponse` 作为 `ApiResponse[Any]` 别名保留 1 版本 | 平滑迁移现有 16 端点 |
| D7 | Alembic 在 P1 末尾引入（先确保模型可创建表） | 避免阻塞模型开发 |
| D8 | EnvelopeMiddleware 仅对 < 1MB JSON 注入 duration_ms | 流式/大响应避免内存压力 |
| D9 | services 层使用 Mock/Real 双模式（与 clients 一致） | 复用 `settings.USE_MOCK` 切换 |
| D10 | 前端路径别名通过 `next.config.js` rewrites 实现 | 双路由并存平滑过渡 |

## 5. 风险与缓解

| 风险 | 缓解措施 |
|---|---|
| Alembic 与现有 `init_db()` 冲突 | P1 末尾引入；保留 `init_db()` 作为 dev fallback |
| 服务层依赖未安装的库（scanpy/cyvcf2/networkx） | Mock/Real 双模式；Real 模式 try-import 降级 |
| 前端 RDKit.js 体积大（~2MB） | dynamic import + ssr:false；首屏不加载 |
| 双路由别名导致 OpenAPI 文档膨胀 | 使用 `include_in_schema=False` 隐藏别名路由 |
| 16 端点改造影响现有前端 | P1 保留 `StandardResponse` 别名；P2 前端适配后再 P4 移除 |
| LLM 成本追踪需要 Redis | 配置项 `LLM_COST_TRACKER_REDIS_URL` 空时降级为内存追踪 |
