### 步骤 B：子系统A 数据接入（Task 4）

**目标**：`/api/v1/data/upload|parse|quality` 端到端可用，支持 RNA-seq/scRNA-seq/VCF/FASTA 四种数据类型。

#### B1. Parser 服务

**`backend/app/services/__init__.py`** — 空文件

**`backend/app/services/parser/__init__.py`** — 导出 `parse_dataset`

**`backend/app/services/parser/base.py`**
- 抽象基类 `Parser`：`async def parse(self, dataset, db) -> dict` 抽象方法
- 工厂函数 `async def parse_dataset(dataset, db) -> dict`：根据 `dataset.data_type` 路由到对应 parser
  - `RNA_SEQ` → `RnaSeqParser`
  - `SCRNA_SEQ` → `ScRnaSeqParser`
  - `WES` → `VcfParser`
  - `FASTA` → `FastaParser`
  - `GENE_REPORT` → 返回简单摘要
- 返回统一结构：`{summary: {rows, cols, ...}, quality_metrics: {...}}`

**`backend/app/services/parser/rna_seq.py`**
- 类 `RnaSeqParser(Parser)`
- 用 `pandas.read_csv` 读取（支持 csv/tsv，sep 自动检测）
- 计算：行数（基因数）、列数（样本数）、缺失率、表达量分布（mean/median/std）、低表达基因比例
- 返回 `{summary: {genes, samples, top_genes}, quality_metrics: {missing_rate, low_expression_ratio, ...}}`

**`backend/app/services/parser/scrna.py`**
- 类 `ScRnaSeqParser(Parser)`
- 用 `scanpy.read_10x_h5` / `read_mtx` 读取
- 标准流程：`sc.pp.filter_cells`、`sc.pp.filter_genes`、`sc.pp.normalize_total`、`sc.pp.log1p`、`sc.pp.highly_variable_genes`、`sc.pp.pca`、`sc.pp.neighbors`、`sc.tl.umap`、`sc.tl.leiden`
- 计算 QC：`n_genes_by_counts`、`total_counts`、`pct_counts_mt`（线粒体比例）
- 差异表达：`sc.tl.rank_genes_groups`（每个 cluster 的 marker genes）
- 返回：`{summary: {n_cells, n_genes, n_clusters, top_markers_per_cluster}, quality_metrics: {median_genes, median_counts, median_mt_pct}}`
- 异常处理：文件不存在或格式错误返回明确错误

**`backend/app/services/parser/vcf.py`**
- 类 `VcfParser(Parser)`
- 用 `cyvcf2.VCF` 读取（如 cyvcf2 不可用则降级到纯文本解析作为容错）
- 统计：变异总数、SNV/INDEL 比例、染色体分布、TS/Tv 比值、含 ClinVar 注释的变异
- 返回：`{summary: {total_variants, snv_count, indel_count, ts_tv_ratio}, quality_metrics: {per_chrom, pass_rate}}`

**`backend/app/services/parser/fasta.py`**
- 类 `FastaParser(Parser)`
- 用 `Bio.SeqIO.parse` 读取
- 统计：序列数、总长度、平均长度、GC 含量、N50
- 返回：`{summary: {sequence_count, total_length, avg_length, gc_content}, quality_metrics: {n50, length_distribution}}`

#### B2. Nextflow 管道

**`nextflow/nextflow.config`**
- 默认 profile：`docker`、`standard`
- 配置 `process.executor = 'local'`、`docker.enabled = true`
- 内存/_cpu 默认值

**`nextflow/scrna_pipeline.nf`**
- 输入：`params.input`（10x h5 文件路径）、`params.output`
- process：`QC_FILTER`、`NORMALIZE_PCA`、`CLUSTER_UMAP`、`DIFF_EXP`
- 输出：`{output}/annotated.h5ad`、`{output}/markers.csv`、`{output}/qc_report.html`

**`nextflow/rna_seq_pipeline.nf`**
- 输入：`params.input`（counts matrix）、`params.samples`（样本元数据）
- process：`QUANTIFY`、`DIFF_EXPRESSION`（用 DESeq2 容器）
- 输出：`{output}/deseq2_results.csv`

**`nextflow/variant_annotation.nf`**
- 输入：`params.vcf`、`params.reference`
- process：`ANNOTATE`（用 VEP 或 SnpEff 容器）、`FILTER`
- 输出：`{output}/annotated.vcf`、`{output}/summary.json`

#### C2. 分析器服务

**`backend/app/services/analyzer/__init__.py`** — 空文件

**`backend/app/services/analyzer/target_identifier.py`** — 子系统B 核心
- 类 `TargetIdentifier(db)`
- `async def discover(project_id, dataset_id=None, tier="fast_screen", hypothesis_id=None) -> dict`：
  1. 查询项目的 datasets（按 dataset_id 过滤或全部）
  2. 从 datasets 中提取变异/差异基因（从 `parsed_summary`）
  3. 调 `VariantClient.query_batch()` 注释变异
  4. 调 `GeneClient.query()` 查询基因信息
  5. 调 `KnowledgeGraph.get_neighbors()` 扩展 PPI 网络
  6. 计算每个靶点的 `confidence_score`（基于：变异致病性 + 表达差异 + 网络中心性）
  7. 分配 `evidence_grade`（已获批药物→I，指南推荐→II，临床试验→III，其他→IV）
  8. 写入 `Target` 表（hypothesis_id 关联）
  9. 返回：`{targets: [{gene_symbol, gene_name, evidence_grade, confidence_score, source, variant_info, annotation, pathway, analysis_tier}], count, tier}`
- 分级处理：
  - `fast_screen`: 仅规则引擎（变异→基因映射），无 LLM 调用，<5s
  - `deep_insight`: 调 LLM 生成深度分析报告，含机制解释

**`backend/app/services/analyzer/drug_repurposer.py`**
- 类 `DrugRepurposer(db)`
- `async def repurpose(target) -> dict`：
  1. 调 `ChemblClient.find_approved_drugs(target.gene_symbol)`
  2. 对每个候选药物，用 RDKit 计算 `molecular_weight`、`logp`、类药性
  3. 评分：批准状态 + 适应症匹配 + 类药性
  4. 返回：`{candidates: [{name, chembl_id, smiles, max_phase, indication, druglikeness_score}], count}`

**`backend/app/services/analyzer/evidence_chain.py`**
- 类 `EvidenceChainBuilder(db)`
- `async def build(target) -> dict`：
  1. 收集证据源：ClinVar 注释、COSMIC 记录、ChEMBL 活性数据、ClinicalTrials.gov 试验、PubMed 文献（可选）
  2. 每条证据：`{source, evidence_type, grade, citation, summary}`
  3. 构建 DAG 结构：`{nodes: [...], edges: [...], root: target.gene_symbol}`
  4. 返回完整证据链

**`backend/app/services/analyzer/molecule_designer.py`** — P2
- 类 `MoleculeDesigner(db)`
- `async def design(payload: dict) -> dict`：
  - **P0 框架**：返回提示信息"分子设计需 P2 启用 DeepChem"
  - **P2 实现**：用 `deepchem.models.GraphConvModel` 预测性质，生成分子骨架
  - 接收：`{target_id, smiles(种子), constraints(分子量/logp/氢键)}`
  - 返回：`{designed_molecules: [{smiles, predicted_activity, properties}], model_info}`
- `def assess_druglikeness(smiles: str) -> dict`：**同步函数**
  - 用 RDKit 计算 Lipinski 五规则：MW≤500、logP≤5、HBD≤5、HBA≤10
  - 计算：`{mw, logp, hbd, hba, rotatable_bonds, tpsa, passes_rule_of_five, violations}`
  - 异常处理：无效 SMILES 返回 `{error}`

**`backend/app/services/analyzer/network_modeler.py`** — P2 框架
- 类 `NetworkModeler(db)`
- `async def analyze_ppi(gene_list: List[str]) -> dict`：
  - P0 框架：调 `KnowledgeGraph.get_neighbors` 简单扩展
  - P2 实现：用 `torch_geometric.nn.GraphSAGE` 训练节点嵌入，识别关键 hub 节点
  - 返回：`{nodes: [...], edges: [...], hub_genes: [...], embedding_dim}`

**`backend/app/services/llm/orchestrator.py`** — 分级路由核心
- 类 `LLMOrchestrator(db, llm_client)`
- `async def route(message, project_id, tier, user) -> dict`：
  1. 根据 `tier` 选择策略
  2. `fast_screen`：直接调 `llm_client.chat()`（无 RAG），cost≈$0.01
  3. `deep_insight`：调 `RAGEngine.retrieve()` + `KnowledgeGraph` + `llm_client.chat()`，cost≈$0.5
  4. 记录 `AnalysisJob`（含 cost_usd, duration_sec, tier）
  5. 返回：`{answer, tier, cost_usd, duration_sec, model, references, code}`
- `async def full_analysis(message, project_id, tier, user) -> dict`：
  1. 复现 Sid 团队流程：提问→文献检索→假设→分析框架→运行→报告
  2. 步骤：解析意图 → 调 `TargetIdentifier` 或 `DrugRepurposer` → 调 LLM 生成报告 → 生成 Python 分析代码
  3. 返回：`{report, charts: [...], code, references, hypothesis, conclusion}`

  - 解析 LIMS 标准格式 payload：`{experiments: [{name, type, result, ...}]}`
  - 批量创建 `Experiment` 记录
  - 返回：`{count, imported_ids, errors}`

**`backend/app/services/experiment/feedback_loop.py`** — 干湿闭环核心
- 类 `FeedbackLoop(db)`
- `async def apply_feedback(experiment) -> dict`：
  1. 比对 dry prediction（experiment.config 中的预测值）vs wet result（experiment.result）
  2. 计算误差：`{mae, rmse, direction_match}`
  3. 触发模型权重更新（P3 框架：调 `federated_learning.update_weights()`）
  4. 标记 `experiment.feedback_applied = True`
  5. 创建下一迭代实验建议：`{next_iteration, suggested_adjustments}`
  6. 返回：`{feedback: {error_metrics, model_updated, next_iteration}}`

## Assumptions & Decisions

1. **默认 Mock 模式**：所有外部 API（OpenAI/MyGene/MyVariant/ChEMBL/DiffDock/ClinicalTrials.gov）在 `USE_MOCK=true` 时返回预置数据，无需任何真实 key 即可端到端运行。
2. **P2/P3 框架策略**：DeepChem/PyG/Flower/PySyft 相关代码以 try/except 包裹，import 失败时返回"需启用 P2/P3"提示，不阻塞主流程。
3. **Nextflow 模拟模式**：`NextflowRunner` 默认返回模拟结果（不实际执行 nextflow），通过 `EXECUTE_NEXTFLOW=true` 环境变量切换真实执行。
4. **文件存储**：P0 用本地 `/data/uploads/` 路径（docker volume 挂载），MinIO 集成留作 P2。
5. **前端 SSR 安全**：Plotly 用 `dynamic(import, {ssr: false})` 避免 Next.js SSR 报错。
6. **测试用 SQLite in-memory**：避免测试依赖 postgres，conftest 用 sqlite + aiosqlite。
7. **种子数据**：5 个角色的演示用户密码统一为 `demo123456`，方便演示。
8. **不修改已存在的 endpoints**：所有 service 严格按反推的接口签名实现，不改 endpoints 代码。
9. **不修改已存在的 models**：service 输出严格对齐已存在的 ORM 字段。
