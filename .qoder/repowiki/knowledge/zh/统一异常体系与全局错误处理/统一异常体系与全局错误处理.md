本仓库在后端 FastAPI 层建立了完整的统一异常体系，通过自定义异常基类、领域异常子类、全局异常处理器以及中间件协作，实现一致的 HTTP 响应信封与结构化错误码。

## 1. 系统/框架
- FastAPI：利用 exception_handler 机制注册全局处理器。
- loguru：所有异常均记录结构化日志（warning / exception），并附带 request_id。
- Starlette ASGI 中间件：自定义 EnvelopeMiddleware 在请求链路中注入 X-Request-ID、X-Response-Time-ms，并对成功响应体中的 meta 字段追加 duration_ms。

## 2. 核心文件与包
- backend/app/core/exceptions.py：定义 AppException 基类及具体异常类型，提供 _envelope / _error_envelope 构造器，并通过 register_exception_handlers(app) 注册三类处理器。
- backend/app/main.py：应用工厂 create_app() 中调用 register_exception_handlers(app)，并挂载 EnvelopeMiddleware。
- 各 API 路由模块（如 auth.py、data.py、efficacy.py、hypotheses.py、molecules.py、privacy.py）直接 raise 上述异常。
- backend/app/schemas/*.py：Pydantic model 的 model_validator 中抛出 ValueError，由 FastAPI 内置校验器转为 RequestValidationError，再被全局处理器捕获。

## 3. 架构与约定
### 3.1 异常层次
AppException (default: 500 INTERNAL_ERROR)
├── ValidationError → 400 VALIDATION_ERROR
├── UnauthorizedError → 401 UNAUTHORIZED
├── ForbiddenError → 403 FORBIDDEN
├── NotFoundError → 404 NOT_FOUND
├── ConflictError → 409 CONFLICT
├── GuardrailBlockedError → 422 GUARDRAIL_BLOCKED
├── RateLimitedError → 429 RATE_LIMITED
└── UpstreamError → 502 UPSTREAM_ERROR
每个子类仅设置 default_status_code 与 default_code，业务消息通过构造参数传入。

### 3.2 全局处理器
- AppException：按 status_code 返回，success=false，error={code,message,details}，meta.request_id 透传；≥500 用 logger.exception，其余 logger.warning。
- RequestValidationError：400 + VALIDATION_ERROR，details 包含原始 errors 列表。
- Exception（兜底）：500 + INTERNAL_ERROR，message 固定为“服务器内部错误，请联系管理员”。

### 3.3 响应信封
成功路径由 ApiResponse[Data] schema 包装，结构为 {success, data, meta}；异常路径由 _error_envelope 生成 {success=false, error, meta}。EnvelopeMiddleware 对 200 JSON 且含 meta 的响应自动注入 meta.duration_ms。

### 3.4 追踪与可观测性
- X-Request-ID：客户端可传入，否则自动生成 hex UUID，写入 scope headers，供依赖注入 get_request_id 使用。
- X-Response-Time-ms：每次响应头携带耗时毫秒数。
- 所有异常日志均带 request_id，便于跨服务关联。

## 4. 开发者应遵循的规则
1. 业务异常一律继承 AppException，不要直接 raise Exception 或裸字符串。
2. 选择最贴近语义的子类：参数问题用 ValidationError，鉴权问题用 UnauthorizedError/ForbiddenError，资源不存在用 NotFoundError，并发冲突用 ConflictError，外部依赖失败用 UpstreamError。
3. 不要在 handler 里 try/except 后再 return JSONResponse——让全局处理器统一封装。
4. Pydantic 校验逻辑放在 model_validator 中 raise ValueError，交由 FastAPI 转 RequestValidationError，避免手写重复代码。
5. 需要附加调试信息时通过 details 字典传递，前端只展示 message，后端日志保留完整 details。
6. 上游调用（LLM、ChEMBL、MyGene 等）失败统一包装为 UpstreamError，保持 502 语义一致。
7. 不要绕过中间件：如需流式响应，注意 EnvelopeMiddleware 对流式分片不做重写，这是预期行为。

该体系覆盖了从输入校验、鉴权、业务规则到外部依赖的全链路错误场景，并以统一信封和结构化错误码对外暴露，配合 loguru 与 request_id 形成可观测的错误闭环。
