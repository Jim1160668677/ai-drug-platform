## 1. 文档概述

### 1.1 目的

本文档描述「精准药物设计系统」（Precision Drug Design，以下简称 PDD）从代码到生产的完整部署流程，覆盖开发环境一键启动、生产环境高可用部署、CI/CD 自动化、数据库迁移、监控告警、故障排查、回滚与升级等运维场景。

### 1.2 适用对象

- **运维工程师 / DevOps 工程师**：负责集群部署、监控、变更
- **后端开发工程师**：负责本地环境搭建、数据库迁移、CI 配置
- **前端开发工程师**：负责前端构建、Nginx 配置
- **架构师 / 技术负责人**：评估部署架构、容量规划

### 1.3 前置技能

| 技能 | 等级 | 用途 |
| --- | --- | --- |
| Docker / Docker Compose | 熟练 | 容器编排、镜像构建 |
| Linux Shell | 熟练 | 服务器运维、脚本编写 |
| PostgreSQL / TimescaleDB | 了解 | 数据库管理、备份恢复 |
| Nginx | 了解 | 反向代理、HTTPS 配置 |
| GitHub Actions | 了解 | CI/CD 流程定制 |
| YAML / Markdown | 熟练 | 配置文件、文档维护 |

## 2. 系统架构总览

### 2.1 服务拓扑（11 个服务）

PDD 平台采用 Docker Compose 编排，共 11 个服务，分为 5 层：

| 层 | 服务 | 镜像 | 角色 |
| --- | --- | --- | --- |
| 接入层 | nginx | nginx:alpine | 反向代理、HTTPS 终止、负载均衡 |
| 应用层 | frontend | 自建（Node 20） | Next.js 14 SSR |
| 应用层 | backend | 自建（Python 3.11） | FastAPI REST API |
| 应用层 | worker | 自建（Python 3.11） | arq 异步任务消费 |
| 计算层 | nextflow | nextflow/nextflow:24.04.0 | 生信工作流引擎 |
| 计算层 | flower-server | python:3.11-slim | 联邦学习服务（profile=phase3，按需启动） |
| 数据层 | postgres | timescale/timescaledb:latest-pg15 | 主关系库 + 时序库 |
| 数据层 | redis | redis:7-alpine | 缓存、消息队列 |
| 数据层 | chromadb | chromadb/chroma:latest | 向量检索（分子/基因嵌入） |
| 数据层 | neo4j | neo4j:5-community | 知识图谱（基因-药物-疾病关系） |
| 数据层 | minio | minio/minio:latest | 对象存储（原始数据、报告） |

### 2.2 端口映射表

| 服务 | 容器端口 | 主机端口 | 用途 | 暴露范围建议 |
| --- | --- | --- | --- | --- |
| nginx | 80 | 80 | HTTP 入口 | 生产开放（或经 LB） |
| frontend | 3000 | 3000 | Next.js dev server | 仅开发开放 |
| backend | 8000 | 8000 | FastAPI | 仅内网/VPC |
| postgres | 5432 | 5432 | PostgreSQL | 仅内网 |
| redis | 6379 | 6379 | Redis | 仅内网 |
| chromadb | 8000 | 8001 | Chroma API | 仅内网 |
| neo4j | 7474 / 7687 | 7474 / 7687 | HTTP UI / Bolt | 仅内网 |
| minio | 9000 / 9001 | 9000 / 9001 | S3 API / Console | 仅内网 |
| nextflow | — | — | 无对外端口 | — |
| flower-server | 8080 | 8080 | 联邦学习 | 仅内网（phase3） |

> **生产环境原则**：仅 nginx 的 80/443 对公网开放，其余端口仅在 VPC 内部可达。

## 3. 环境要求

### 3.1 硬件要求

| 资源 | 开发环境 | Staging | 生产环境（最小） | 生产环境（推荐） |
| --- | --- | --- | --- | --- |
| CPU | 4 核 | 8 核 | 16 核 | 32 核 |
| 内存 | 8 GB | 16 GB | 32 GB | 64 GB |
| 磁盘 | 50 GB SSD | 200 GB SSD | 500 GB SSD | 1 TB NVMe + 对象存储扩展 |
| 网络 | 100 Mbps | 1 Gbps | 1 Gbps | 10 Gbps |

> **说明**：
> - RDKit、scanpy、chromadb 在加载大分子库或单细胞数据时内存峰值可达 4–8 GB，建议预留余量。
> - Nextflow 工作流产物与原始数据上传文件主要存放在 MinIO，磁盘容量按业务增长 +30% 缓冲规划。
> - 联邦学习（phase3）启用时额外占用 4 GB 内存。

### 3.2 软件要求

| 软件 | 最低版本 | 推荐版本 | 备注 |
| --- | --- | --- | --- |
| Docker Engine | 24.0 | 24.0+ | 必须 |
| Docker Compose | v2.20 | v2.26+ | 使用 v2 语法（`docker compose`） |
| Node.js | 20.0 | 20.x LTS | 前端构建与本地调试 |
| Python | 3.11 | 3.11.x | 后端本地调试 |
| Conda / Mamba | 24.0 | latest | 用于本地非容器化开发环境 |
| Git | 2.40 | latest | 代码版本管理 |
| Make | 4.0 | latest | 执行 Makefile 命令 |

### 3.3 网络要求

**对外访问（出站）**：

| 目标 | 端口 | 用途 |
| --- | --- | --- |
| `api.openai.com` | 443 | LLM 调用（USE_MOCK=false 时） |
| `mygene.info` / `myvariant.info` | 443 | 基因/变异注释 |
| `www.ebi.ac.uk/chembl` | 443 | ChEMBL 化合物数据 |
| `clinicaltrials.gov` | 443 | 临床试验数据 |
| `integrate.api.nvidia.com` | 443 | DiffDock NIM 推理 |
| GitHub Container Registry | 443 | CI 拉取/推送镜像 |

**对内开放（入站）**：

- 80 / 443：HTTP/HTTPS 入口
- 22：SSH 运维通道（建议限定源 IP）
- 仅在 VPC 内：5432 / 6379 / 8000 / 3000 / 9000 / 7474 / 7687 / 8080

**防火墙示例（ufw）**：

```bash
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw allow 22/tcp           # SSH
sudo ufw allow 80/tcp           # HTTP
sudo ufw allow 443/tcp          # HTTPS
sudo ufw enable
```

## 4. 快速开始（开发环境）

### 步骤 1：克隆代码

```bash
git clone <your-repo-url> AI药物
cd AI药物
```

确认目录结构：

```
AI药物/
├── backend/           # FastAPI 后端
├── frontend/          # Next.js 前端
├── docker/            # docker-compose 与配置
│   ├── docker-compose.yml
│   ├── nginx/nginx.conf
│   └── postgres/init.sql
├── nextflow/          # 生信工作流
├── data/              # 数据卷（运行时生成）
├── docs/              # 文档
├── Makefile
├── .env.example
└── environment.yml
```

### 步骤 2：配置 .env

```bash
cp .env.example .env
```

**必填项（开发环境）**：

| 变量 | 示例值 | 说明 |
| --- | --- | --- |
| `JWT_SECRET_KEY` | 随机 32 字符 | **必填**，生产必改 |
| `POSTGRES_PASSWORD` | `pdd_secret` | 数据库密码 |
| `NEO4J_PASSWORD` | `neo4j_secret` | Neo4j 密码 |
| `MINIO_SECRET_KEY` | `pdd_minio_secret` | MinIO 密钥 |
| `API_KEY_ENCRYPTION_KEY` | Fernet 密钥 | 加密用户存储的 LLM Key（生成方法见 §5.4） |

**开发 Mock 模式（默认）**：保留 `USE_MOCK=true`、`OPENAI_API_KEY=`（留空），系统将以内置 Mock 数据运行，无需外部 API。

```bash
# 生成 JWT 密钥
openssl rand -hex 32

# 生成 Fernet 密钥
python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
```

将上述输出写入 `.env`。

### 步骤 3：启动服务

```bash
make up
# 或
docker compose -f docker/docker-compose.yml --env-file .env up -d
```

启动后访问地址：

- 前端：http://localhost
- API 文档：http://localhost/docs
- MinIO 控制台：http://localhost:9001
- Neo4j 浏览器：http://localhost:7474

### 步骤 4：初始化数据库

容器启动时 `backend` 服务会自动执行 `alembic upgrade head`（见 `docker-compose.yml` 的 `command` 字段）。如需手动执行：

```bash
make migrate
# 或
docker compose -f docker/docker-compose.yml exec backend alembic upgrade head
```

`docker/postgres/init.sql` 仅在 PostgreSQL 数据卷首次创建时执行，包含：

- 启用 `uuid-ossp` / `pgcrypto` / `timescaledb` 扩展
- 创建 `clinical_timeline` 与 `efficacy_timeseries` 两张 hypertable
- 创建 `audit_log_append`（append-only 审计日志，触发器禁止 UPDATE/DELETE）

### 步骤 5：加载种子数据

```bash
make seed
# 或
docker compose -f docker/docker-compose.yml exec backend python -m app.db.seed
```

种子数据包含示例项目、靶点、分子、用户等，便于前端联调。

### 步骤 6：验证服务健康

```bash
# 查看所有服务状态
make ps

# 后端健康检查
curl http://localhost:8000/health
# 期望返回：
# {"status":"healthy","app":"precision-drug-design","version":"1.0.0","mock_mode":true,"env":"development"}

# 通过 nginx 访问
curl http://localhost/health
curl http://localhost/
```

各服务健康检查命令：

```bash
docker compose -f docker/docker-compose.yml ps
# 关注 STATUS 列，应为 Up (healthy)
```

### 步骤 7：访问应用

打开浏览器：

- **生产路径（推荐）**：http://localhost
- **直连前端**：http://localhost:3000
- **直连后端 API**：http://localhost:8000/api/v1
- **Swagger 文档**：http://localhost/docs
- **ReDoc 文档**：http://localhost/redoc

> 开发模式下前端挂载源码卷并启用 `npm run dev` 热重载；后端 `uvicorn --reload` 自动重启。

## 6. 生产部署

### 6.1 镜像构建

#### 6.1.1 后端镜像

`backend/Dockerfile` 基于 `python:3.11-slim`，安装生物信息学库所需的系统依赖（`build-essential`、`libffi-dev`、`libgomp1`、`libgl1-mesa-glx` 等）。构建时利用 `requirements.txt` 单独 COPY 触发缓存层：

```bash
docker build -t precision-drug-backend:1.0.0 ./backend
```

#### 6.1.2 前端镜像（多阶段）

`frontend/Dockerfile` 三阶段构建：

1. **deps**：`npm ci` 安装依赖（利用 npm 缓存）
2. **builder**：`npm run build` 生成 `.next/standalone` 产物
3. **runner**：仅拷贝 `public`、`.next/standalone`、`.next/static`，以非 root `nextjs` 用户运行

构建参数：

```bash
docker build \
  --build-arg NEXT_PUBLIC_API_BASE_URL=https://api.example.com/api/v1 \
  --build-arg NEXT_PUBLIC_USE_MOCK=false \
  -t precision-drug-frontend:1.0.0 ./frontend
```

#### 6.1.3 缓存策略

## 7. CI/CD 流程

### 7.1 Workflow 总览

| Workflow | 文件 | 触发条件 | 主要任务 |
| --- | --- | --- | --- |
| Backend CI | `.github/workflows/backend-ci.yml` | push 到 `main`/`develop`，PR | Lint、Test、Coverage、Build Image |
| Frontend CI | `.github/workflows/frontend-ci.yml` | push 到 `main`/`develop`，PR | Lint、Type Check、Build |
| Deploy | `.github/workflows/deploy.yml` | push 到 `main` 或手动 workflow_dispatch | 构建并推送镜像、创建 Release、健康检查、可选回滚 |

### 7.2 Backend CI 详解

**Job: test**

1. Checkout 代码
2. Setup Python 3.11（缓存 pip）
3. `pip install -r requirements.txt`
4. `ruff check app/ tests/`
5. FastAPI app 加载自检：`from app.main import app`
6. `pytest tests/ --cov=app --cov-fail-under=80`（覆盖率阈值 80%）
7. 上传 `coverage.xml` 与 `TEST_REPORT.md` 作为 artifact（保留 14 天）

**Job: build**

- 仅当 `push` 到 `main` 时执行
- 使用 `docker/build-push-action` 构建镜像
- 启用 GHA 缓存（`cache-from: type=gha`）
- 镜像 tag：`precision-drug-backend:latest`（仅本地构建，未推送）

**环境变量**：

```yaml
APP_ENV: test
DATABASE_URL: sqlite+aiosqlite:///./test.db
SECRET_KEY: ci-test-secret-key-not-for-production-32chars
API_KEY_ENCRYPTION_KEY: ''   # CI 中允许为空
ACCESS_TOKEN_EXPIRE_MINUTES: '60'
```

### 7.3 Frontend CI 详解

1. Setup Node.js 20（缓存 npm）
2. `npm ci`
3. `npm run lint`
4. `npx tsc --noEmit`
5. `npm run build`（注入 `NEXT_PUBLIC_API_BASE_URL`）
6. 上传 `.next` 作为 artifact（保留 7 天）

### 7.4 Deploy Workflow 详解

**触发方式**：

- 自动：push 到 `main` 且 `backend/`、`frontend/`、`docker-compose.yml` 或 workflow 自身变更
- 手动：`workflow_dispatch`，可选 `environment`（`staging` / `production`）与 `rollback` 布尔

**部署 Job 步骤**：

1. 生成 `release_tag`：`YYYYMMDD-HHMMSS-<short-sha>`
2. 构建并推送 backend/frontend 镜像到 `${{ secrets.DOCKER_REGISTRY }}`，双 tag（时间戳 + `latest`）
3. 创建 GitHub Release（production 为正式版，其他为 prerelease）
4. Slack 通知（通过 `SLACK_WEBHOOK`）

**Health Check Job**：

- `sleep 30` 等待服务启动
- 轮询 10 次 `${{ secrets.HEALTH_CHECK_URL }}/health`
- 返回 200 视为成功，否则失败

### 7.5 部署到服务器（手动 SSH 方式）

```bash
# 1. 登录目标服务器
ssh deploy@prod-server

# 2. 拉取最新镜像
cd /opt/pdd
docker compose -f docker/docker-compose.yml \
               -f docker/docker-compose.prod.yml \
               --env-file .env.prod pull

# 3. 滚动更新
docker compose -f docker/docker-compose.yml \
               -f docker/docker-compose.prod.yml \
               --env-file .env.prod up -d backend worker frontend

# 4. 执行数据库迁移
docker compose -f docker/docker-compose.yml exec backend alembic upgrade head

# 5. 健康检查
curl -f http://localhost/health || exit 1
```

### 7.6 Required Secrets

| Secret | 用途 |
| --- | --- |
| `DOCKER_REGISTRY` | 镜像仓库地址（如 `ghcr.io/your-org`） |
| `DOCKER_USERNAME` / `DOCKER_PASSWORD` | 镜像仓库凭证 |
| `HEALTH_CHECK_URL` | 部署后健康检查 URL |
| `SLACK_WEBHOOK` | 部署通知 |

## 8. 数据库管理

### 8.1 Alembic 迁移流程

配置文件 `backend/alembic.ini`，迁移脚本目录 `backend/alembic/`，`env.py` 从 `settings.DATABASE_URL` 动态读取连接串。

**生成新迁移**：

```bash
make migrate-new m="add_user_table"
# 或
docker compose -f docker/docker-compose.yml exec backend \
  alembic revision --autogenerate -m "add user table"
```

**应用迁移**：

```bash
make migrate
# 或
docker compose -f docker/docker-compose.yml exec backend alembic upgrade head
```

**查看当前版本**：

```bash
docker compose -f docker/docker-compose.yml exec backend alembic current
docker compose -f docker/docker-compose.yml exec backend alembic history
```

**回滚**：

```bash
# 回退一个版本
docker compose -f docker/docker-compose.yml exec backend alembic downgrade -1

# 回退到指定版本
docker compose -f docker/docker-compose.yml exec backend alembic downgrade <revision>

# 回退到初始状态（危险！）
docker compose -f docker/docker-compose.yml exec backend alembic downgrade base
```

**注意事项**：

- `alembic upgrade head` 已被 `docker-compose.yml` 写入 backend 启动命令，容器重启会自动执行
- 自动生成的迁移必须人工审核 `upgrade()` / `downgrade()` 是否对称
- 生产环境迁移前必须在 staging 验证，并备份 `pg_dump`

### 8.2 备份策略

#### 8.2.1 逻辑备份（pg_dump）

## 11. 故障排查

### 11.1 服务启动失败

**端口冲突**：

```bash
# 查看端口占用
netstat -tulpn | grep -E ':(80|3000|5432|6379|8000) '
# Linux
sudo lsof -i :80

# 释放端口
sudo kill -9 <PID>
```

**依赖未就绪**：

backend 启动命令已包含 `alembic upgrade head`，若 postgres 未健康，`depends_on: condition: service_healthy` 会阻止启动。如仍失败：

```bash
# 检查 postgres 状态
docker compose -f docker/docker-compose.yml ps postgres
docker compose -f docker/docker-compose.yml logs postgres
```

### 11.2 数据库连接失败

```bash
# 1. 容器内测试
docker compose -f docker/docker-compose.yml exec backend \
  python -c "import asyncio; from app.db.session import async_session; \
  asyncio.run(async_session().execute('SELECT 1'))"

# 2. 检查连接串
docker compose -f docker/docker-compose.yml exec backend env | grep DATABASE_URL

# 3. psql 直连
docker compose -f docker/docker-compose.yml exec postgres \
  psql -U pdd -d precision_drug -h localhost -c "SELECT 1;"

# 4. 检查 pg_hba.conf
docker compose -f docker/docker-compose.yml exec postgres cat /var/lib/postgresql/data/pg_hba.conf
```

常见原因：

- `DATABASE_URL` 中 host 应为容器服务名 `postgres`，本地直连改为 `localhost`
- 密码包含特殊字符需 URL 编码
- PostgreSQL 未启动或数据卷损坏

### 11.3 前端构建失败

```bash
# 1. 查看构建日志
docker compose -f docker/docker-compose.yml logs frontend

# 2. 进入容器手动构建
docker compose -f docker/docker-compose.yml exec frontend sh
npm install
npm run build

# 3. 清理缓存
docker compose -f docker/docker-compose.yml exec frontend rm -rf .next node_modules
docker compose -f docker/docker-compose.yml restart frontend
```

常见原因：

- `NEXT_PUBLIC_API_BASE_URL` 未设置
- Node 版本不匹配（要求 20+）
- `.next` 卷权限问题（删除匿名卷重建：`docker compose down -v && docker compose up -d`）

### 11.4 LLM 调用失败

```bash
# 1. 检查 USE_MOCK 与 OPENAI_API_KEY
docker compose -f docker/docker-compose.yml exec backend env | grep -E "USE_MOCK|OPENAI"

# 2. 测试 OpenAI 连通性
docker compose -f docker/docker-compose.yml exec backend \
  python -c "from openai import OpenAI; c=OpenAI(); print(c.models.list())"

# 3. 检查 Fernet 密钥
docker compose -f docker/docker-compose.yml exec backend env | grep API_KEY_ENCRYPTION_KEY
```

常见原因：

- `USE_MOCK=true` 但期望真实调用 → 改为 `false`
- `OPENAI_API_KEY` 为空或过期
- `API_KEY_ENCRYPTION_KEY` 与加密时使用的密钥不一致，导致用户存储的 Key 无法解密
- 超过 `FAST_SCREEN_MAX_DURATION_SEC` / `DEEP_INSIGHT_MAX_DURATION_SEC` 阈值

### 11.5 内存不足（OOM）

```bash
# 查看容器内存使用
docker stats

# 查看系统 dmesg（OOM Killer）
dmesg | grep -i "killed process"
```

应对：

- 增加 `deploy.resources.limits.memory`
- 限制 `uvicorn --workers` 数量
- scanpy / chromadb 处理大数据时分区加载
- 关闭非必要服务（如 phase3 的 flower-server）

### 11.6 磁盘空间不足

## 12. 回滚机制

### 12.1 镜像回滚

```bash
# 1. 查看历史镜像 tag
docker images precision-drug-backend --format "{{.Tag}} {{.CreatedAt}}"

# 2. 修改 docker-compose.prod.yml，使用旧 tag
#   image: precision-drug-backend:20260704-120000-abc1234

# 3. 拉取旧镜像并重启
docker compose -f docker/docker-compose.yml \
               -f docker/docker-compose.prod.yml \
               --env-file .env.prod pull
docker compose -f docker/docker-compose.yml \
               -f docker/docker-compose.prod.yml \
               --env-file .env.prod up -d backend worker frontend

# 4. 验证
curl http://localhost/health
```

### 12.2 数据库回滚

```bash
# 1. 查看迁移历史
docker compose -f docker/docker-compose.yml exec backend alembic history

# 2. 回退到指定版本
docker compose -f docker/docker-compose.yml exec backend \
  alembic downgrade <previous_revision>

# 3. 验证表结构
docker compose -f docker/docker-compose.yml exec postgres \
  psql -U pdd -d precision_drug -c "\dt"
```

> 警告：`downgrade` 会丢弃新增数据。生产环境回滚前必须 `pg_dump` 备份。

### 12.3 配置回滚

将 `.env` 纳入 Git 版本管理（注意脱敏），或使用 Vault 维护历史版本：

```bash
# 从 Git 恢复 .env
git log --oneline -- .env
git show <commit>:.env > .env.rollback
# 对比差异
diff .env .env.rollback
# 应用
cp .env.rollback .env
docker compose -f docker/docker-compose.yml up -d
```

### 12.4 回滚验证清单

- [ ] `/health` 返回 200，版本号符合预期
- [ ] 关键 API 冒烟测试通过（登录、项目列表、靶点查询）
- [ ] 前端首页可正常加载
- [ ] 数据库表结构与回滚目标版本一致
- [ ] Worker 正常消费任务（`docker logs pdd-worker`）
- [ ] 监控告警恢复正常

## 13. 升级流程

### 13.1 版本兼容性检查

- 阅读 `CHANGELOG.md` 与 `docs/技术架构文档.md` 的版本变更章节
- 检查 `requirements.txt` / `package.json` 的依赖升级
- 确认数据库 schema 变更范围（`alembic/versions/`）
- 评估向后兼容性：API 路由、配置项、密钥格式

### 13.2 数据库迁移测试

```bash
# 1. 在 staging 环境复制生产数据
docker compose -f docker/docker-compose.yml exec -T postgres \
  pg_dump -U pdd -Fc precision_drug > prod.dump
# 上传到 staging
docker compose -f docker/docker-compose.yml exec -T postgres \
  pg_restore -U pdd -d precision_drug -c prod.dump

# 2. 执行迁移并验证
docker compose -f docker/docker-compose.yml exec backend alembic upgrade head
docker compose -f docker/docker-compose.yml exec backend alembic check

# 3. 执行回滚测试，确认 downgrade 可逆
docker compose -f docker/docker-compose.yml exec backend alembic downgrade -1
docker compose -f docker/docker-compose.yml exec backend alembic upgrade head
```

### 13.3 灰度发布

借助 Nginx weighted upstream：

```nginx
upstream backend_api {
    server backend-new:8000 weight=10;
    server backend-old:8000 weight=90;
}
```

逐步调整权重：90/10 → 50/50 → 10/90 → 100/0。

观察指标：

- 错误率（5xx）
- 平均响应时延
- 业务核心 API 通过率

### 13.4 蓝绿部署

```bash
# 蓝环境（当前生产）
docker compose -f docker/docker-compose.blue.yml --env-file .env.prod up -d

# 绿环境（新版本部署）
docker compose -f docker/docker-compose.green.yml --env-file .env.prod up -d

# 切换流量
docker compose -f docker/docker-compose.yml exec nginx nginx -s reload

# 蓝环境保留 24h 作为快速回滚资源，再清理
docker compose -f docker/docker-compose.blue.yml down
```

## 14. 安全加固

### 14.1 防火墙配置

```bash
# 仅允许办公网段访问 80/443
sudo ufw allow from 10.0.0.0/8 to any port 80
sudo ufw allow from 10.0.0.0/8 to any port 443

# SSH 仅限跳板机
sudo ufw allow from <jumphost-ip> to any port 22

# 默认拒绝
sudo ufw default deny incoming
sudo ufw enable
```

### 14.2 密钥管理

**禁止**将以下密钥提交到 Git：

- `JWT_SECRET_KEY`
- `API_KEY_ENCRYPTION_KEY`
- `OPENAI_API_KEY`
- `MINIO_SECRET_KEY`
- `NEO4J_PASSWORD`
- `POSTGRES_PASSWORD`

推荐方案：

| 方案 | 适用场景 |
| --- | --- |
| HashiCorp Vault | 自建 KMS，统一密钥轮转 |
| AWS Secrets Manager / 阿里云 KMS | 云原生托管 |
| Docker Secrets / Swarm Secrets | 单机或小规模集群 |
| `.env.prod` + 文件权限 600 + 服务器隔离 | 简单部署 |

`.env` 文件权限：

```bash
chmod 600 .env
chown deploy:deploy .env
```

### 14.3 容器安全

#### 14.3.1 非 root 用户

前端 `frontend/Dockerfile` 已使用 `nextjs:nodejs` 非 root 用户运行。后端建议补充：
