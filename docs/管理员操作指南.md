## 1. 概述

### 1.1 文档目的

本指南面向「精准药物设计系统」的管理员用户，介绍管理后台的访问方式、核心功能（用户管理、LLM 配置、审计日志）的标准操作流程，以及常见问题排查方法。本指南不涉及开发与部署细节。

### 1.2 适用对象

| 角色 | 是否可访问管理后台 | 可执行操作 |
| --- | --- | --- |
| 创始人（founder） | ✅ 是 | 全部管理操作 |
| 首席研究员（chief_researcher） | ✅ 是 | LLM 配置管理、审计日志查看 |
| 研究员（researcher） | ❌ 否 | — |
| 医生（doctor） | ❌ 否 | — |
| 数据工程师（data_engineer） | ❌ 否 | — |

### 1.3 前置条件

- 已由系统创始人或上级管理员为您创建账户，并分配 `founder` 或 `chief_researcher` 角色。
- 已知本人登录邮箱与初始密码。
- 浏览器推荐：Chrome / Edge 最新版（需支持现代 JavaScript 与 ES Module）。
- 网络可访问平台前端地址（默认 `http://<服务器IP>:3000`）。

### 1.4 登录方式

1. 在浏览器中打开平台首页。
2. 点击右上角「登录」按钮，输入邮箱与密码。
3. 登录成功后，系统签发 JWT 令牌（有效期 24 小时），前端会自动跳转至工作台。
4. 顶部导航栏出现「管理后台」入口（仅 `founder` / `chief_researcher` 角色可见）。

> ⚠️ 令牌过期后需重新登录。如长时间未操作，系统将自动登出。

## 2. 角色与权限体系

### 2.1 五种角色说明

系统采用基于角色的访问控制（RBAC）模型，共 5 种角色，权限由低到高排列如下：

| 角色标识（系统内） | 角色名称 | 定位 |
| --- | --- | --- |
| `founder` | 创始人 | 系统最高权限，全部数据 + 模型干预 + 紧急决策 |
| `chief_researcher` | 首席研究员 | 全部分析结果 + 部分原始数据 + LLM 配置 |
| `researcher` | 研究员 | 仅可访问分配项目内的数据 |
| `doctor` | 医生 | 临床数据 + 靶点报告只读 |
| `data_engineer` | 数据工程师 | 系统日志 + 数据质量指标 |

> 📌 **说明**：后端枚举中 `chief_researcher` 与 `data_engineer` 的字符串值分别为 `chief` 和 `engineer`，前端在不同位置可能展示任一形式。本指南统一使用完整名称。

### 2.2 权限矩阵

| 权限项 | founder | chief_researcher | researcher | doctor | data_engineer |
| --- | :---: | :---: | :---: | :---: | :---: |
| 原始数据读取（全量） | ✅ | ✅ | — | — | — |
| 分配项目内数据读取 | ✅ | ✅ | ✅ | — | — |
| 临床数据读取 | ✅ | ✅ | — | ✅ | — |
| 数据写入 | ✅ | — | — | — | — |
| 分析结果读取 | ✅ | ✅ | — | — | — |
| 分析结果写入 | ✅ | ✅ | — | — | — |
| 标准分析任务执行 | ✅ | ✅ | ✅ | — | — |
| 模型干预 | ✅ | — | — | — | — |
| 模型配置 | ✅ | ✅ | — | — | — |
| 紧急决策 | ✅ | — | — | — | — |
| 决策建议 | ✅ | ✅ | — | ✅ | — |
| 靶点报告读取 | ✅ | ✅ | — | ✅ | — |
| 审计日志读取 | ✅ | ✅ | — | — | — |
| 系统日志读取 | ✅ | — | — | — | ✅ |
| 数据质量指标 | ✅ | — | — | — | ✅ |
| 系统配置 | ✅ | — | — | — | ✅ |
| 管理后台访问 | ✅ | ✅ | — | — | — |
| 用户管理（增删改） | ✅ | — | — | — | — |

### 2.3 关键规则

- 仅 `founder` 可执行用户管理（查看列表、改角色、启用/禁用）。
- `founder` 角色不可被授予（无法通过界面将任何人提升为 `founder`）。
- `founder` 账户不可被禁用。
- 用户不能修改自己的角色，也不能禁用自己的账户。

## 3. 进入管理后台

### 3.1 访问地址

- **路径**：`/admin`
- **完整 URL 示例**：`http://<服务器IP>:3000/admin`
- 入口：登录后在顶部导航栏点击「管理后台」链接，或直接在地址栏输入上述路径。

### 3.2 访问限制

后台由前端与后端双层校验：

1. **前端校验**：未登录用户访问 `/admin` 会被自动重定向至首页 `/`。
2. **角色校验**：登录用户但角色不属于 `founder` / `chief_researcher` 时，页面将显示「403 — 权限不足」提示卡片。
3. **后端校验**：即便绕过前端，所有管理类 API 均要求对应角色令牌，未授权请求将返回 `403 Forbidden`。

### 3.3 403 处理流程

如访问后台时看到「403 — 权限不足」：

| 可能原因 | 处理方法 |
| --- | --- |
| 当前角色不是 founder / chief_researcher | 联系 founder 为您提升角色 |
| 角色已提升但前端缓存未刷新 | 退出登录并重新登录 |
| 账户被禁用 | 联系 founder 重新启用 |
| 令牌过期 | 重新登录获取新令牌 |

> 💡 提示：刷新页面或清除浏览器缓存后再试一次，多数 403 问题可解决。

### 3.4 操作示例

```
示例 1：首席研究员 Alice 需配置新的 LLM
  1) 在登录页输入 alice@hospital.cn / 密码；
  2) 登录后点击「管理后台」；
  3) 在「大模型 API 设置」卡片中新建配置（详见第 6 章）。
```

## 4. 系统配置概览

进入管理后台后，首个卡片为「系统配置」，集中展示当前运行环境关键信息：

| 配置项 | 说明 | 示例值 |
| --- | --- | --- |
| 运行模式 | 是否启用 Mock 模式（不调用真实外部 API） | Mock / Real |
| API 版本 | 后端接口版本号 | v1.0.0 |
| 当前用户 | 当前登录账户邮箱 | admin@pdd.cn |
| 角色 | 当前用户角色徽章 | founder / chief_researcher |

### 4.1 关键参数说明

- **运行模式 = Mock**：系统不真实调用 LLM、DiffDock、外部生物医学 API，使用模拟数据返回。适用于演示、培训、联调。
- **运行模式 = Real**：调用真实外部服务，需保证 LLM API Key、NVIDIA NIM Key 等已正确配置。
- **API 版本**：当前为 v1.0.0，后续版本升级时管理员需关注兼容性公告。

### 4.2 切换运行模式

运行模式由后端环境变量 `USE_MOCK` 决定，**前端无法直接切换**。如需切换：

1. 评估切换影响范围（建议在维护窗口执行）。
2. 联系运维 / 数据工程师修改后端 `.env` 文件中的 `USE_MOCK` 值。
3. 重启后端服务。
4. 刷新管理后台，确认「运行模式」显示已切换。

> ⚠️ 切换至 Real 模式前，请确认：
> - 至少一个 LLM 配置已激活并通过连通性测试；
> - `JWT_SECRET_KEY`、`API_KEY_ENCRYPTION_KEY` 已设置为生产级强随机值；
> - 外部生物医学 API（mygene、myvariant、ChEMBL 等）网络可达。

## 5. 用户管理

> 本章节操作**仅限 `founder` 角色**。`chief_researcher` 在后台看不到「用户列表」卡片。

### 5.1 查看用户列表

#### 5.1.1 入口

管理后台主页 → 「用户列表 (N)」卡片。

#### 5.1.2 列表字段

| 字段 | 说明 |
| --- | --- |
| 邮箱 | 用户登录邮箱（等宽字体显示） |
| 姓名 | 用户真实姓名 |
| 角色 | 角色徽章 |
| 组织 | 所属机构名称 |
| 状态 | 启用（绿色对勾）/ 禁用（红色叉号） |
| 创建时间 | 用户账户创建日期 |
| 操作 | 角色下拉框 + 启用/禁用按钮 |

#### 5.1.3 筛选与分页

- **按角色筛选**：右上角下拉框选择「研究员 / 医生 / 数据工程师 / 首席研究员」。
- **按状态筛选**：右上角下拉框选择「已启用 / 已禁用」。
- **结果数量**：列表默认加载最多 200 条用户记录（后端 `limit` 上限为 200）。当总用户数超过 200 时，需联系开发扩展分页支持。
- **筛选条件可叠加**：例如同时选择「角色=研究员」+「状态=已禁用」，可查看所有被禁用的研究员。

#### 5.1.4 操作示例

```
示例：查看所有被禁用的医生账户
  1) 在「角色」下拉框选择「医生」；
  2) 在「状态」下拉框选择「已禁用」；
  3) 列表自动刷新，显示符合条件的用户。
```

> 📷 *（此处插入截图：用户列表筛选效果）*

### 5.2 修改用户角色

#### 5.2.1 操作步骤

1. 在用户列表中定位目标用户。
2. 在该行「操作」列的角色下拉框中选择新角色（仅可选择：研究员 / 医生 / 数据工程师 / 首席研究员）。
3. 选择不同角色时系统自动提交，列表立即刷新。

#### 5.2.2 限制规则

| 场景 | 系统行为 |
| --- | --- |
| 修改自己的角色 | 后端返回 400「不能修改自己的角色」 |
| 将用户提升为 founder | 后端返回 400「不能将用户提升为 founder」 |
| 操作 founder 账户 | 列表中 founder 行的操作列显示「—」，无任何操作入口 |
| 输入非法角色值 | 后端返回 400「无效角色: xxx」 |
| 目标用户不存在 | 后端返回 404「用户不存在」 |

#### 5.2.3 操作示例

```
示例：将研究员 bob@lab.cn 提升为首席研究员
  1) 在用户列表中找到 bob@lab.cn；
  2) 在其「操作」列下拉框中选择「首席研究员」；
  3) 列表刷新，bob 的角色徽章变更为「首席研究员」；
  4) 审计日志自动记录该操作。
```

> ⚠️ 提升为首席研究员后，该用户即可访问管理后台并管理 LLM 配置，请谨慎操作。

### 5.3 启用 / 禁用用户

#### 5.3.1 操作步骤

1. 在用户列表中定位目标用户。
2. 点击该行「操作」列的「禁用」或「启用」按钮。
3. 状态列立即更新。

#### 5.3.2 限制规则

| 场景 | 系统行为 |
| --- | --- |
| 禁用自己的账户 | 后端返回 400「不能禁用自己的账户」 |
| 禁用 founder 账户 | 后端返回 400「不能禁用 founder 账户」 |
| 操作 founder 账户 | 列表中 founder 行无操作入口 |
| 操作自己 | 列表中自己所在行无操作入口 |
| 目标用户不存在 | 后端返回 404 |

#### 5.3.3 禁用后的影响

- 被禁用用户的令牌仍可能有效至过期（最长 24 小时），但其后续登录请求会被拒绝。
- 已签发的 JWT 不会主动失效。如需立即清退，建议联系开发实现令牌黑名单。

#### 5.3.4 操作示例

```
示例：临时禁用已离职研究员 carol@lab.cn
  1) 在用户列表中找到 carol@lab.cn；
  2) 点击「禁用」按钮；
  3) 状态列变为「禁用」（红色叉号）；
  4) 审计日志记录：actor=admin, action=user.status.update, entity=user:<id>。
```

## 6. LLM 大模型配置管理

> 本章节操作限 `founder` 与 `chief_researcher` 角色。是管理后台最核心的运维模块。

### 6.1 配置列表与激活机制

#### 6.1.1 列表字段

管理后台 → 「大模型 API 设置」卡片，表格字段如下：

| 字段 | 说明 |
| --- | --- |
| 名称 | 配置名称（如 Agnes、OpenAI、Azure） |
| 访问模式 | API Only / 本地部署 / 代理 |
| 协议 | Chat Completions / Completions / Anthropic Messages |
| 基础 URL | 上游 LLM 服务地址（等宽字体，超长截断） |
| API Key | 脱敏显示（保留前 6 位与后 4 位，中间省略） |
| 测试模型 | 用于连通性测试的模型名称 |
| 状态 | 激活（绿色徽章）/ 未激活（灰色文字）+ 最近测试结果 |
| 操作 | 激活 / 测试 / 编辑 / 删除 图标 |

#### 6.1.2 激活机制

- 系统**任意时刻仅允许一个 LLM 配置处于激活状态**。
- 新建配置时勾选「设为当前激活配置」、或对已有配置点击「激活」图标，其他配置会**自动置为非激活**。
- 系统业务调用 LLM 时，优先使用数据库中激活的配置；若不存在激活配置，则降级使用 `settings` 默认值（Mock 模式下不真实调用）。
- 卡片顶部蓝色提示框始终展示「当前激活：xxx」与测试模型，便于一眼确认生效配置。

### 6.2 新建配置

#### 6.2.1 必填字段

| 字段 | 必填 | 校验规则 |
| --- | :---: | --- |
| 配置名称 | ✅ | 唯一，不可与已有配置重名 |
| 基础 URL | ✅ | 合法 URL，如 `https://apihub.agnes-ai.com/v1` |
| API 密钥 | ✅（新建时） | 字符串，建议 `sk-` 前缀格式 |
| 测试模型 | ✅ | 模型标识符，如 `agnes-2.0-flash` |
| Provider | — | 默认 `openai_compatible`，可填 `agnes` 等 |
| 访问模式 | — | 下拉选择，默认 API Only |
| 上游协议 | — | 下拉选择，默认 Chat Completions |
| 快速筛查模型 | — | 可选，用于快速分析任务 |
| 深度洞察模型 | — | 可选，用于深度分析任务 |
| 温度 | — | 数值 0–2，默认 0.7 |
| 最大 Token | — | 整数 1–32000，默认 2000 |
| 超时（秒） | — | 整数 1–600，默认 60 |
| 描述 | — | 文本，便于识别 |
| 设为当前激活 | — | 复选框，默认不勾选 |

#### 6.2.2 表单校验规则

- 必填字段为空 → 提示「表单校验失败」并定位到对应字段。
- 配置名称重复 → 后端返回 409「配置名称 'xxx' 已存在」。
- 访问模式 / 上游协议值非法 → 后端返回 422「参数无效」。
- 数值字段超范围 → 前端校验阻止提交。

#### 6.2.3 Agnes 预设一键填充

为简化配置流程，卡片右上角提供「Agnes 预设」按钮，点击后自动填充以下值：

| 字段 | 预设值 |
| --- | --- |
| 配置名称 | `Agnes` |
| Provider | `agnes` |
| 访问模式 | API Only |
| 上游协议 | Chat Completions |
| 基础 URL | `https://apihub.agnes-ai.com/v1` |
| 测试模型 | `agnes-2.0-flash` |
| 快速筛查模型 | `agnes-2.0-flash` |
| 深度洞察模型 | `agnes-2.0-flash` |
| 描述 | Agnes AI 大模型 API（API Only / Chat Completions） |
| 设为当前激活 | ✅ 勾选 |

> 💡 使用预设后，**仍需手动填入 API 密钥**，然后点击「保存」。

#### 6.2.4 操作示例

```
示例：使用 Agnes 预设新建配置
  1) 在「大模型 API 设置」卡片右上角点击紫色「Agnes 预设」按钮；
  2) 弹出表单已自动填充名称、URL、模型等；
  3) 在「API 密钥」字段填入从 Agnes 控制台获取的 Key（如 sk-xxxxxxxxxxxx）；
  4) 确认「设为当前激活配置」已勾选；
  5) 点击「保存」；
  6) 列表刷新，新配置显示「激活」徽章。
```

### 6.3 编辑配置

#### 6.3.1 操作步骤

1. 在配置列表中点击目标行的「编辑」（铅笔）图标。
2. 弹出编辑表单，所有字段已填充原值（API 密钥字段为空）。
3. 修改需要变更的字段后点击「保存」。

#### 6.3.2 API Key 留空表示不修改

- 编辑表单中 API 密钥字段**默认为空**，并在标签处提示「（留空表示不修改）」。
- 留空提交时，前端不会传 `api_key` 字段，后端保留数据库原值不变。
- 如需更换 API Key，手动输入新值即可，保存后系统自动用 Fernet 重新加密存储。

> ⚠️ 编辑时 API 密钥字段**不再是必填**。但新建时为必填。

### 6.4 测试连通性

#### 6.4.1 测试入口

管理后台提供两种测试方式：

| 方式 | 操作 | 适用场景 |
| --- | --- | --- |
| 测试指定配置 | 点击列表行内「测试」（闪电）图标 | 验证未激活配置 |
| 测试激活配置 | 在卡片下方输入框旁点击「测试激活配置」按钮 | 验证当前生效配置 |

#### 6.4.2 自定义测试消息

- 卡片下方提供文本输入框，可填写自定义测试消息。
- 不填则默认发送 `ping` 作为测试消息。
- 测试请求会以 `temperature=0.1`、`max_tokens=50` 发送，避免消耗过多配额。

#### 6.4.3 结果解读

测试完成后，结果区域以彩色面板展示：

| 字段 | 含义 |
| --- | --- |
| 成功 / 失败标识 | 绿色对勾 = 成功，红色叉号 = 失败 |
| 消息 | 「连接成功（X.XXXs）」或具体错误信息 |
| 模型 | 实际响应模型名（来自 LLM 返回） |
| 响应 | LLM 回复内容（截断至 500 字符） |
| 耗时 | 请求总耗时（秒） |

常见错误原因：

| 错误信息 | 可能原因 |
| --- | --- |
| HTTP 401 | API Key 错误或已失效 |
| HTTP 403 | Key 无该模型权限或访问受限 |
| HTTP 404 | 基础 URL 错误或路径拼接异常 |
| HTTP 429 | 触发上游限流，稍后重试 |
| ConnectError / Timeout | 网络不通或超时设置过短 |
| 暂不支持的协议 | 当前测试仅支持 Chat Completions / Completions 协议 |

#### 6.4.4 测试结果持久化

每次测试后，系统会自动更新该配置的 `last_test_at` / `last_test_success` / `last_test_message` 三个字段。列表「状态」列下方会显示「最近测试通过 / 最近测试失败」标识，便于事后追溯。

### 6.5 激活 / 删除配置

#### 6.5.1 激活配置

1. 在配置列表中找到未激活的目标配置。
2. 点击「激活」（电源）图标。
3. 系统自动将该配置置为激活，其他配置同步置为非激活。
4. 卡片顶部蓝色提示框更新为新激活的配置名。

> 💡 已激活的配置行不会显示「激活」图标，避免重复操作。

#### 6.5.2 删除配置

1. 在配置列表中点击目标行的「删除」（垃圾桶）图标。
2. 系统弹出确认框「确认删除配置 'xxx' 吗？此操作不可恢复。」。
3. 点击「确认」即可删除。

#### 6.5.3 不可删除激活配置

- **删除按钮对激活配置无效**：前端会拦截并提示「不能删除当前激活的配置，请先切换到其他配置」。
- 后端同样返回 400「不能删除当前激活的配置，请先切换到其他配置」。
- 正确流程：先激活其他配置 → 再删除原配置。

### 6.6 加密存储说明

#### 6.6.1 加密方案

- 所有 LLM API Key 在入库前均使用 **Fernet 对称加密**（来自 `cryptography` 库）加密。
- 密文以 `enc:` 前缀标识，便于解密时识别。
- 数据库中存储形如 `enc:gAAAAABm...` 的密文字符串，原始 Key 不落库。
- 列表展示时仅返回脱敏形式（保留前 6 位与后 4 位）。

#### 6.6.2 加密密钥

- 加密密钥由后端环境变量 `API_KEY_ENCRYPTION_KEY` 配置。
- **生产环境必须**设置为 32 字节 base64 字符串（通过 `Fernet.generate_key()` 生成）。
- 若未配置或配置非法，系统降级为明文存储并打印警告日志（**严重安全风险**）。

> ⚠️ **密钥保管注意事项**：
> - 切勿将密钥提交到代码仓库或日志中。
> - 密钥一旦丢失，所有已加密的 API Key 将无法解密，需全部重新录入。
> - 密钥泄露后，应立即轮换密钥并重新加密所有 API Key。

## 7. 审计日志

> 本章节操作限 `founder` 与 `chief_researcher` 角色（需 `audit:read` 权限）。

### 7.1 日志字段说明

管理后台主页底部「审计日志 (N)」卡片默认展示最近 50 条日志，每条记录包含以下字段：

| 字段 | 说明 | 示例 |
| --- | --- | --- |
| 时间 | 操作发生时间（本地时区） | 2026/7/5 14:30:21 |
| 用户 | 操作者邮箱 + 角色 | admin@pdd.cn / founder |
| 操作 | 操作类型徽章 | user.role.update / llm.config.create |
| 资源 | 被操作实体类型 + ID 前 8 位 | user:a1b2c3d4 |
| 详情 | 操作详情文本 | 角色 researcher → chief |

> 📌 后端日志记录还包含 **IP 地址** 与 **User-Agent** 字段（前端表格未展示，但 API 返回）。如需查看，请通过 API 直接查询。

### 7.2 查询与筛选

#### 7.2.1 前端筛选

- 管理后台默认拉取最近 50 条日志，不支持前端筛选条件。
- 如需更复杂查询（按时间范围、按用户、按操作类型），请使用后端 API。

#### 7.2.2 后端 API 查询

接口：`GET /api/v1/audit/logs`

支持的查询参数：

| 参数 | 类型 | 说明 |
| --- | --- | --- |
| `actor` | string | 按操作者邮箱筛选 |
| `action` | string | 按操作类型筛选（如 `user.role.update`） |
| `entity` | string | 按实体类型筛选（如 `user` / `llm_config`） |
| `skip` | int | 跳过条数（默认 0） |
| `limit` | int | 返回条数（1–500，默认 100） |

请求示例（需附带 Authorization 头）：

```
GET /api/v1/audit/logs?actor=admin@pdd.cn&action=user.role.update&limit=50
Authorization: Bearer <JWT>
```

#### 7.2.3 操作示例

```
示例：导出某管理员近 30 天所有用户管理操作
  1) 通过 API 查询：GET /api/v1/audit/logs?actor=admin@pdd.cn&limit=500；
  2) 在返回结果中筛选 action 以 "user." 开头的记录；
  3) 复制到 Excel 或数据库工具做进一步分析。
```

### 7.3 日志保留策略建议

系统当前未自动清理审计日志，长期运行后将占用数据库空间。建议策略：

| 保留周期 | 用途 | 实施方式 |
| --- | --- | --- |
| 近 90 天 | 日常运维查询 | 数据库热存储 |
| 90 天–1 年 | 合规审计 | 归档至对象存储（MinIO） |
| 1 年以上 | 长期留档 | 压缩备份至离线介质 |

巡检建议：

- 每月检查 `audit_log` 表行数增长趋势。
- 单表超过 100 万行时考虑分区或归档。
- 归档操作必须由数据工程师在维护窗口执行，并记录归档批次号。

> ⚠️ 审计日志**不可篡改**：任何对日志的修改本身也应被记录（如使用触发器或应用层双写）。删除日志需经创始人书面授权。

## 8. 常见问题 FAQ

### Q1：登录失败，提示「邮箱或密码错误」怎么办？

**可能原因**：
- 邮箱或密码输入错误（注意大小写、空格）。
- 账户已被禁用。
- 密码曾被管理员重置但未收到通知。

**处理方法**：
1. 确认邮箱拼写无误，检查输入法是否为英文模式。
2. 联系 founder 确认账户状态是否为「启用」。
3. 如忘记密码，请走「忘记密码」流程或联系 founder 重置（见 Q7）。

### Q2：访问管理后台提示「403 — 权限不足」但我是首席研究员？

**可能原因**：
- 浏览器缓存了旧的角色信息（角色刚被提升时常见）。
- 令牌已过期但前端未刷新。

**处理方法**：
1. 退出登录并重新登录获取新令牌。
2. 强制刷新页面（Ctrl+F5）。
3. 若仍失败，联系 founder 确认数据库中角色值是否为 `chief`。

### Q3：LLM API Key 测试失败怎么办？

**常见错误与排查**：

| 错误信息 | 排查方向 |
| --- | --- |
| HTTP 401 Unauthorized | API Key 错误、过期或被吊销，重新申请新 Key |
| HTTP 403 Forbidden | Key 未开通该模型权限，联系供应商开通 |
| HTTP 404 Not Found | 基础 URL 错误，检查末尾是否包含 `/v1` |
| HTTP 429 Too Many Requests | 触发限流，等待 1–2 分钟后重试 |
| ConnectError | 服务器无法访问外网，检查代理 / 防火墙 |
| Timeout | 调大「超时（秒）」字段（建议 ≥ 30） |
| 暂不支持的协议 | 当前测试仅支持 Chat Completions / Completions，Anthropic 协议暂不支持测试 |

### Q4：为什么无法禁用某个用户？

**可能原因**：
- 该用户是 founder（系统禁止禁用 founder 账户）。
- 该用户就是你自己（系统禁止禁用自己的账户）。
- 该用户在前端列表中显示「—」表示不可操作。

**处理方法**：
- 如确需禁用 founder，先将其角色改为其他角色（但需另一个 founder 操作；若仅一位 founder，需开发介入）。
- 自禁用需求：请其他 founder 代为操作。

### Q5：删除 LLM 配置时提示「不能删除当前激活的配置」？

**原因**：系统禁止删除激活配置，避免业务无可用 LLM。

**处理方法**：
1. 先在列表中找到另一个配置，点击「激活」图标。
2. 待原配置变为「未激活」状态后，再点击「删除」图标即可。

### Q6：审计日志卡片显示「暂无审计日志」？

**可能原因**：
- 系统刚部署完成，尚未产生任何管理操作。
- 当前用户没有 `audit:read` 权限（被前端隐藏）。
- 后端审计日志服务异常。

**处理方法**：
1. 执行一次管理操作（如修改一个用户角色），再刷新后台查看是否产生日志。
2. 若仍为空，联系数据工程师检查 `audit_log` 表与后端日志服务。
3. 确认当前用户角色包含 `audit:read` 权限（founder / chief_researcher）。

### Q7：忘记密码怎么办？

**处理流程**：
1. 在登录页点击「忘记密码」链接（如已配置）按流程自助重置。
2. 若未启用自助重置，联系 founder 提交重置请求。
3. founder 通过后端管理工具或数据库脚本重置密码（需开发协助）。
4. 收到新初始密码后，**首次登录立即修改密码**。

> ⚠️ 出于安全考虑，密码以 bcrypt 哈希存储，**任何人都无法查看原始密码**，只能重置。

### Q8：管理后台响应慢，加载卡顿怎么办？

**可能原因**：
- 用户数或审计日志量过大（默认拉取 200 用户 / 50 日志）。
- 后端数据库慢查询。
- 网络延迟高。

**处理方法**：
1. 先使用角色 / 状态筛选缩小用户列表范围。
2. 通过 API 分页查询审计日志（避免一次性拉取 500 条）。
3. 联系数据工程师检查后端响应耗时与数据库索引。
4. 必要时安排维护窗口优化数据库或扩容服务。

### Q9（补充）：如何确认当前生效的 LLM 是哪个？

- 管理后台「大模型 API 设置」卡片**顶部蓝色提示框**始终展示当前激活配置名与测试模型。
- 也可调用 `GET /api/v1/llm-configs/active` 接口确认。
- 若返回 `use_default: true`，表示无激活配置，系统使用 `settings` 默认 LLM（Mock 模式下不真实调用）。

### Q10（补充）：JWT 令牌过期后如何处理？

- 令牌默认有效期 24 小时（`JWT_ACCESS_TOKEN_EXPIRE_MINUTES=1440`）。
- 过期后所有 API 返回 401，前端会自动重定向至登录页。
- 重新登录即可，无需特殊处理。

## 9. 安全最佳实践

### 9.1 密码策略

- **复杂度**：至少 12 位，包含大写、小写、数字、特殊字符各 1 个。
- **更换周期**：建议每 90 天更换一次；发现疑似泄露立即更换。
- **不可复用**：最近 5 次历史密码不可重复使用（建议后端启用密码历史）。
- **存储**：所有密码以 bcrypt 哈希存储，任何人都无法逆向查看。
- **初始密码**：管理员创建账户后通过安全渠道（如加密邮件）下发，**首次登录强制改密**。

### 9.2 API Key 轮换

- **轮换周期**：建议每 90 天轮换一次 LLM API Key。
- **轮换流程**：
  1. 在 LLM 供应商控制台生成新 Key。
  2. 管理后台编辑对应配置，填入新 Key，保存。
  3. 点击「测试」验证新 Key 可用。
  4. 在供应商控制台吊销旧 Key。
- **泄露应急**：发现 Key 泄露立即吊销 + 替换 + 排查审计日志中异常请求。
- **加密密钥轮换**：`API_KEY_ENCRYPTION_KEY` 建议每年轮换一次，需走专门脚本重新加密所有 Key（联系开发）。

### 9.3 审计日志监控

- **每周巡检**：检查异常时间的操作、非常用 IP 的登录、批量数据导出等异常行为。
- **告警规则建议**（联系开发实现）：
  - 短时间内多次登录失败（≥ 5 次/分钟）。
  - founder 账户被修改或禁用尝试。
  - LLM 配置在非工作时间被修改。
  - 审计日志被删除或修改。
- **日志完整性**：定期备份审计日志至独立存储，防止主库被攻破后日志同步丢失。

### 9.4 定期巡检清单

| 巡检项 | 频率 | 负责人 | 检查要点 |
| --- | --- | --- | --- |
| 用户清单核对 | 每月 | founder | 离职用户是否已禁用，角色是否合理 |
| LLM 配置核对 | 每月 | founder / chief | 是否有未使用的废弃配置 |
| 审计日志归档 | 每月 | 数据工程师 | 超 90 天日志是否已归档 |
| API Key 轮换 | 每 90 天 | founder | LLM Key 是否到期前已更换 |
| 加密密钥轮换 | 每年 | founder + 开发 | `API_KEY_ENCRYPTION_KEY` 是否已轮换 |
| 环境变量核对 | 每季度 | 数据工程师 | `JWT_SECRET_KEY` 等是否仍为默认值（必改） |
| 角色权限复核 | 每季度 | founder | 是否存在过度授权 |
| 备份恢复演练 | 每半年 | 数据工程师 | 数据库与配置可成功恢复 |

### 9.5 通用安全要求

- **不在公共网络**下访问管理后台；如必须，使用 VPN。
- **不共享账户**：每个管理员使用独立账户，便于审计追溯。
- **不截图泄露**：管理后台截图分享前必须打码用户邮箱、API Key 脱敏片段。
- **及时退出**：离开工位前点击右上角「退出登录」。

## 10. 紧急联系与升级流程

### 10.1 紧急情况分类

| 级别 | 定义 | 示例 |
| --- | --- | --- |
| P0 — 严重 | 系统不可用、数据泄露、安全事件 | LLM 配置全部失效、API Key 疑似泄露、数据库被攻破 |
| P1 — 高 | 核心功能不可用、影响多人 | 管理后台无法登录、用户管理功能异常 |
| P2 — 中 | 单点功能异常、有规避方案 | 某项筛选失效、审计日志查询慢 |
| P3 — 低 | 体验问题、不影响业务 | UI 显示错位、文案错误 |

### 10.2 联系矩阵

| 级别 | 第一联系人 | 第二联系人 | 响应时效 |
| --- | --- | --- | --- |
| P0 | 创始人（手机） | 数据工程师（24×7） | 15 分钟内 |
| P1 | 创始人 | 首席研究员 | 1 小时内 |
| P2 | 首席研究员 | 数据工程师 | 4 小时内 |
| P3 | 数据工程师 | — | 1 个工作日内 |

> 📌 实际联系方式（电话、邮箱、企业 IM 账号）由团队维护一份独立的《紧急联系人通讯录》，本文档不直接列出，请向创始人索取。

### 10.3 升级流程

#### 10.3.1 安全事件应急流程（P0）

1. **立即止血**：
   - 若为 API Key 泄露，立即登录管理后台编辑对应配置更换 Key。
   - 若为账户被攻破，立即禁用该账户。
   - 若为数据库被攻破，立即联系数据工程师切断外网访问。
2. **保留证据**：导出审计日志、保存服务器访问日志、记录操作时间线。
3. **通知**：按联系矩阵通知创始人，必要时同步至合规部门。
4. **复盘**：48 小时内召开复盘会议，输出事件报告并落实改进措施。

#### 10.3.2 功能故障升级流程（P1/P2）

1. **复现**：尝试用不同账户、不同浏览器复现问题。
2. **截图记录**：保存报错截图、浏览器控制台日志、操作时间。
3. **提单**：通过企业 IM 或工单系统提交故障单，注明级别与影响范围。
4. **跟踪**：负责人接单后按响应时效回复，每日同步进展直至闭环。

#### 10.3.3 变更审批流程

涉及以下变更时，需创始人书面（含 IM 截图）审批后执行：

- 修改 `JWT_SECRET_KEY` 或 `API_KEY_ENCRYPTION_KEY`。
- 切换 `USE_MOCK` 模式。
- 删除任何 LLM 配置。
- 批量禁用用户（≥ 5 个）。
- 删除或归档审计日志。
- 数据库结构变更。
